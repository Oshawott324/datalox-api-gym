Datalox Env Data Proof v0 Post-Training Runbook

This runbook describes the handoff contract for a small post-training workflow review. The goal is not to claim reliable lift from 13 tasks. The goal is to check whether the world package, tool-message transcripts, tool-env eval rows, and verifier contract fit a normal model-team rollout and training stack.

Source readiness is gated by source-datasets.manifest.json and task-source-gate.csv. Current FlowCyto and Molecule Biology rows are smoke data until recaptured from the approved public flowCore and NCBI sources.

13 eval rows in exports/eval.seed.jsonl
7/3/3 train/dev/test smoke split
7 train-only final-answer SFT smoke rows
7 train-only tool-evidence SFT rows
7 train-only system/user/assistant/tool SFT rows

Non-Negotiable Boundary

The baseline model and the SFT target must be the same trainable open-weight base model. A closed API model can be useful as a reference inference run, but it does not answer whether this package supports SFT unless that same model can be fine-tuned.

Do not train on dev/test rows. The current package keeps target answers for every task so verifiers can be smoke-tested, but the exported SFT files contain train rows only.
Tool-env eval is the primary training proof. eval.seed.jsonl is a context-eval smoke file because it preloads replay observations in the prompt. The environment claim is tested by eval.tool_env.seed.jsonl, where the agent must use domain tools and accumulate evidence through actions.

Default Smoke Model

Use Qwen/Qwen3-1.7B as the default public smoke target unless the post-training collaborator prefers another trainable small model. The Hugging Face model card lists it as Apache-2.0, 1.7B parameters, and 32,768-token context, with Transformers support.

The model choice is replaceable. The invariant is not Qwen specifically; the invariant is same open-weight base before and after SFT.

Step 0: Package State

The handoff package now contains a runnable world preview at runnable-world/. It creates task workspaces, exposes local deterministic tools through ./datalox_tool, accepts ./submit_answer answer.json, writes trajectory.jsonl, and writes verifier_result.json.

Verified local state: runnable-world oracle answers pass and bad answers fail for FASTQ QC and molecule primer validation. The wider seed still has 13 tasks, a 7/3/3 smoke split, 7 train-only tool-message SFT rows, and no source-gate claim that the seed is lift-grade data.

Step 1: Plug Into The Runnable World

The first collaborator task is world fit, not JSONL review. A harness should initialize a task, enter the generated workspace, let the agent read README.md and task.json, call ./datalox_tool, write answer.json, and call ./submit_answer answer.json.

The useful feedback is whether this workspace interface maps to the team's existing agent runner, whether it needs HTTP/MCP/Python reset-step wrapping, and whether trajectory.jsonl contains enough information for training and debugging.

Step 2: Map The World Contract

The current preview contract is workspace-first: init_task creates a workspace, ./datalox_tool returns observations, ./submit_answer returns verifier result and reward, and the run directory contains durable trajectory artifacts. If the collaborator's stack wants reset/step, wrap this contract instead of treating JSONL rows as the environment. 

# Target harness shape, implemented in the model team's normal agent runner:
reset(task_id) -> initial system/user messages and available tools
tools() -> model-visible tool schemas
step(tool_call) -> deterministic tool observation
finalize(answer) -> verifier result and reward fields
export_messages() -> system/user/assistant/tool transcript

A successful baseline from the collaborator's side should report pass/fail by split and task family, meaningful failure modes, replay or environment misses, model id, model revision when available, inference stack, hardware, wall time, and estimated inference cost.

The older exports/eval.tool_env.seed.jsonl rows are secondary eval descriptors. The runnable world is the environment preview.

Step 3: Prepare Train Rows

Use exports/sft.tool_messages.seed.jsonl as the primary SFT source for collaborator review. It contains standard system, user, assistant, and tool turns: every assistant tool call is followed by the matching tool observation, and the final assistant turn is the verifier-passing answer.

Keep exports/sft.tool_evidence.seed.jsonl for Datalox audit/provenance. Use exports/sft.seed.chat.jsonl only as a final-answer format smoke file. If a stack wants only the message array without Datalox provenance columns, the adapter can strip each row to {"messages": row.messages}.

The full files keep provenance fields such as source_answer_path, verifier_spec_path, observation_path, and split_metadata_path. Use those fields for audit, not as target text.

Step 4: Run Minimal LoRA SFT

Use the collaborator's normal SFT stack: TRL, Axolotl, Unsloth, LLaMA Factory, or internal tooling. Datalox does not require a custom trainer. The minimum run should use:

  • Base model: Qwen/Qwen3-1.7B or the agreed replacement.
  • Primary training file: exports/sft.tool_messages.seed.jsonl.
  • Audit/provenance file: exports/sft.tool_evidence.seed.jsonl.
  • Format-smoke training file: exports/sft.seed.chat.jsonl if the stack cannot yet train tool turns.
  • Training rows only: 7 rows in this smoke split.
  • Output: LoRA adapter or merged model with the base model id recorded.
This SFT run is intentionally tiny. It validates formatting, tool_steps masking, adapter loading, and held-out eval mechanics. It is not a credible model-improvement claim by itself.

Step 5: Re-Run The Same Tool-Env Eval

Serve the SFT adapter or merged model through the same inference path, then run the exact same tool-env eval rows: 

eval.tool_env.seed.jsonl
  -> same agent harness
  -> same allowed tools
  -> SFT model or base+LoRA adapter
  -> verifier report by split/family

Compare verifier pass rate on dev/test only. Train-pass improvement is useful for debugging the SFT setup, but it is not the public metric.

Decision Gate

  • Pass: the collaborator can load the rows, train a LoRA, re-run eval, and produce a split-aware report without asking how the environment works.
  • Weak pass: formatting works, but no held-out improvement appears. Next action is to collect the 30/10/10 lift-grade set.
  • Fail: rows are too long, target format is wrong, or verifier rewards do not match useful model behavior. Fix the export contract before collecting more tasks.

What This Proves

This proves whether Datalox can package agent-native scientific tasks into a normal post-training workflow: tool-env eval rows, captured tool-use trajectories, train-only SFT rows, deterministic verifier results, and a same-base before/after comparison. The context rows are still useful smoke artifacts, but the environment proof depends on the tool-env path. It does not prove reliable model lift until the dataset is scaled beyond this smoke split.

The next dataset scale target is 30 train, 10 dev, 10 test. The more credible public demo target is 80 train, 20 dev, 20 test.
