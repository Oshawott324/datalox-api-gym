Datalox concept demo for AI-ready lab automation

After programmable lab control, agents need verifiable practice.

Replay/eval is the front door: capture what an agent did through lab APIs, score whether it was correct, then reuse verified traces as resettable dry-run environments.

Call objective: spend less than 3 minutes on the concept, then learn where this fits relative to UniteLabs' roadmap and customer pain.
My read on UniteLabs

UniteLabs is building the programmable control plane for labs.

Control

Connect instruments, expose SDK/API surfaces, and let workflows command devices directly.

Workflow

Run lab-as-code workflows with deployment, logs, artifacts, human-in-the-loop steps, and data capture.

AI-ready

Make instruments and workflows understandable enough for AI systems to plan, call, and adapt.

Datalox should not compete with that layer. It should sit above it: replay, eval, audit, and dry-run environments for agents using programmable lab surfaces.

Why another layer

Real execution is the source of truth, but not the right place for every agent attempt.

Expensive

Instrument time, reagents, consumables, and API quotas make repeated trial-and-error costly.

Slow

Physical workflows have latency. Agents need many attempts, but labs cannot wait for every exploratory rollout.

Side-effectful

Liquid transfers, incubations, measurements, and device state changes cannot be replayed physically for free.

The question is when an agent should pay for reality, and when a verified record can support debugging, audit, eval, or dry-run.

Datalox ladder

Replay/eval is the hook. Resettable environments are the compounding layer.

1. Capture

Record exact tool/API I/O, observations, state fingerprints, and side effects from real or sandboxed runs.

2. Score

Use verifiers to check final state and invariants, so the trace becomes eval data rather than a passive log.

3. Reuse

Turn verified traces into regression tests and resettable dry-run environments before touching instruments.

This is workflow-level simulation: tools, observations, state ledgers, constraints, and verifiers. It is not a claim to perfectly simulate biology or chemistry.
Concrete artifact

Structured tool I/O plus state-based verification.

{"tool":"liquid_handler.aspirate",
 "args":{"source":"source_plate:A1","volume_ul":50},
 "result":{"ok":true,"observed_volume_ul":50}}

{"tool":"plate_reader.read_absorbance",
 "args":{"plate":"assay_plate","wells":["B1"]},
 "result":{"ok":true,"values":{"B1":0.82}}}
{
  "verifier": "plate_qc_transfer_and_readout",
  "ok": true,
  "checks": [
    "source_not_overdrawn",
    "target_volume_recorded",
    "readout_after_dispense",
    "decision_matches_observation"
  ]
}

The agent cannot pass by writing a plausible final answer. The verifier checks environment state and workflow invariants.

Discovery

The questions for the call.

Current pain

  • Is the bottleneck still connectivity, or are replay and validation already painful?
  • When workflows fail mid-run, what state is captured today?
  • What is still hard to reconstruct after a failed run?

Future agent workflows

  • What would make an AI-generated workflow safe enough to execute?
  • Would dry-runs or resettable synthetic worlds be useful before physical runs?
  • Which lab workflow shape would be best for a tiny non-proprietary prototype?
