One-Sentence Positioning

Datalox reconstructs traditional human-facing workflows as agent-native domain environments, then exports replayable training/eval data for SFT, preference training, eval, and eventually RL.

This is intentionally not "Agentic RL infra first." RL is valuable, but it is expensive: rollouts are slow, verifiers are bottlenecks, and reward signals are often sparse. The near-term product value is agent-native workflow environments plus installable snapshot environments that produce clean trajectories for export.

Current Implemented Surface

The repo now has a concrete first path for model-server users instead of only fixture metadata: 

OpenAI-compatible model endpoint, including vLLM
  -> datalox run
  -> replay fixture world
  -> datalox_run.v1
  -> datalox export sft
  -> sft_frame.v1 JSONL

This is the practical bridge to SFT-first training teams. Datalox does not replace vLLM, SGLang, LiteLLM, verl, or a lab's internal training stack. It provides the declared replay world, grounded transcript, and export row. 

datalox run \
  --fixture-set support-triage-basic@2026-05.0 \
  --base-url http://127.0.0.1:8000/v1 \
  --model Qwen/Qwen2.5-7B-Instruct \
  --api-key token \
  --prompt "Resolve this support triage task." \
  --out runs/support-triage-qwen \
  --json

datalox export sft \
  --run runs/support-triage-qwen \
  --out exports/support-triage-qwen.sft.jsonl \
  --json

Implemented now: OpenAI-compatible model adapter, replay tool runtime, datalox_run.v1, datalox run, sft_frame.v1, and datalox export sft. Still next: published serious fixture sets, completion frame export, preference pair export, direct verl adapter, and live domain environment rollout proofs.

Who Uses What

Rollout / Environment Owner

Runs agents against real tools, domain MCPs, APIs, or app worlds. Uses Datalox to author private snapshots, package replay bundles, and publish versioned fixture worlds.

Training / Eval Consumer

Wants installable fixture sets and clean versioned exports, not capture infrastructure. Starts with SFT and preference data; may later use reward/eval rows and online RL.

Fixture / Data Provider

Curates approved snapshot fixture packs and fixture sets so downstream teams can train, evaluate, and compare agents without live tool credentials.

Authoring Path vs Consumption Path

Authoring path  (rollout owner - needs live tools)
live domain MCP / API / app world
  -> Datalox snapshot authoring path
  -> tool_io_record.v1
  -> replay_bundle.v1
  -> fixture pack
  -> fixture set

Consumption path  (training/eval consumer - no live tools)
versioned API/MCP snapshot fixture set install
  -> replay runtime
  -> datalox_run.v1
  -> sft_frame.v1 now
  -> completion_frame.v1 + preference_pair.v1 next
  -> SFT / preference training / reward modeling / eval / regression
  -> online RL later, only with runtime-backed reset/step

Important boundary: replay mode never invents observations and never falls back to live tools. Unseen calls return structured replay misses.

Training Consumer Order

OrderConsumerDatalox OutputBoundary
1 SFT-first teams sft_frame.v1 now, completion_frame.v1 next Use successful runs first. Recovery trajectories can be added when labeled. Do not blindly clone failures.
2 Preference training preference_pair.v1 next Success > recovery > failure under the same task, all backed by replay evidence.
3 Reward modeling / eval reward_record.v1, verifier provenance, reference scheme Datalox records evidence and reference schemes. Training teams own reward design.
4 Online RL transition_frame.v1, runtime-backed rollout evidence Requires reset/step worlds, rollout throughput, and verifier capacity.

Reward Boundary

Datalox should not own canonical reward design. Reward is task-specific and belongs to the team doing training. Datalox should preserve enough evidence so the reward can be recomputed, audited, or replaced. 

Datalox owns:
  reward source
  reward version
  verifier command/result
  evidence refs
  optional reference reward scheme

Training team owns:
  final reward function
  weights
  normalization
  clipping
  training objective

Why SFT First

Zheng's feedback changes the priority order but not the product boundary: online Agent RL is expensive, slow to roll out, and often suffers from sparse reward. Many teams will get value sooner from environments that generate many clean trajectories for SFT and preference training.

Datalox should therefore optimize the first mile: 

fixture/world breadth
  -> stable tool observations
  -> model-visible transcript
  -> exportable SFT/completion/preference rows
  -> model-team-owned optimization

This is not a retreat from RL. It is the path that produces useful data before a team can afford large on-policy rollout loops.

Environment Strategy

A single deep environment proves seriousness, but breadth is likely better for improving general agent capability. The recommended strategy is: 

one credible deep-enough proof
  -> many shallow/medium verified sub-environments
  -> SFT/preference exports at scale
  -> runtime-backed online RL later

For bio, this means an environment family rather than one perfect simulator. 

bio-agent-world@2026-05

  flowcyto-qc-report

AppWorld Lesson

AppWorld is a strong reference for what a whole environment looks like: simulated users, app databases, APIs, task generation, state-based evaluation, checkpointing, logs, and an MCP/API surface.

The lesson is not "clone AppWorld." The practical Datalox proof is: 

AppWorld MCP/API server
  -> Datalox authors a versioned snapshot
  -> replay_bundle.v1
  -> fixture set
  -> SFT/preference/eval exports
