Snapshot Environment And Training Data Roadmap

Datalox provides resettable, verifiable API worlds, live domain MCP environments in sibling repos, and training/eval exports from those worlds. Recording is an authoring path for private snapshots. Runtime integrations come later and stay separate from pure replay.

Updated: 2026-05-22. Related docs: local-to-server-engine-plan.html, agentic-rl-layer-map.md, replay-bundle-schema.md.

Stable boundary: Datalox API Gym provides versioned API/MCP snapshot worlds; it does not build generic task worlds. Sibling Datalox domain MCP repos may build constrained scientific worlds; this roadmap defines how Replay snapshots, verifies, serves, and exports those worlds and external runtimes.

Mode boundary: datalox replay --fixture is pure replay. It must not start Docker, CyberGym, E2B, Modal, OpenClaw, a browser runtime, or live upstream services. Commands that start an executable environment are runtime-backed record, rollout, or verification commands, not plain replay.

Public wording rule: describe adoption by workflow, not by customer resources. Use SFT-first teams, teams starting from SFT, or limited rollout budget. The public product claim is environment first: versioned API worlds produce SFT and preference data as immediate outputs; online RL is a later consumer when reset/step runtimes and rollout throughput exist.

External Runtime

Owns containers, services, binaries, filesystem state, browsers, vulnerable targets, network policy, queueing, and compute scheduling.

Datalox Adapter

Owns adapter lifecycle integration, tool-I/O interception, replay-bundle packing, evidence verification, fixture activation, and grounded exports. It does not own runtime isolation, scheduling, image construction, or security semantics.

Agent Harness

Owns model choice, scaffold, prompts, context management, rollout loop, verifier calls, and final training/eval consumption.

Why This Is Later

Heavy runtimes solve a different bottleneck from Datalox's first wedge: agent-native workflow surfaces plus replay evidence. Docker, CyberGym-style environments, and internal clusters answer: 

Can I create and verify a full executable environment?

Datalox first answers:

Can I expose the workflow as agent-native tools and replay the agent-visible evidence?

Most teams already have some runtime, even if it is imperfect. Replacing it is risky because it touches security, secrets, networking, CI, GPUs, queues, and billing. A replay layer can be adopted earlier because it sits beside the runtime and makes evals reproducible without forcing infrastructure migration.

Online agent RL is expensive because rollout is slow, verifier throughput is limited, and reward is often sparse. The first practical training consumer is usually SFT or preference training over high-quality trajectories. Online RL remains important, but it should come after replay, API worlds, exports, and runtime-backed reset/step loops are stable.

Paper-Informed Delta

Five systems surveyed. Each defines "world" differently. The common pattern is not Docker everywhere. The common pattern is: 

state substrate + action API + observation renderer + verifier/reward + rollout logging

Docker is one implementation of "state substrate + reset." It is not the contract itself.

Agent-World (arXiv:2604.18292) — database + executable toolset world

Environment = structured database + many executable Python tools. Built by mining web data into databases, generating tool interfaces from MCP specs and PRDs, validating with compiler/tests, synthesizing tasks from tool dependency graphs, running in Python sandbox, collecting traces with LLM rubrics and verifier scripts, training with GRPO. Key detail: the world is not one tool. It is a stateful database plus many tools where tools read and write shared state and return structured observations.

EvoEnv (arXiv:2605.14392) — environment as code

Environment = a Python class. Not MCP-first or Docker-first. The class implements generate(seed, difficulty), prompt_generate(), process(response), and scorer(parsed_response). Admitted only after: Python parses, methods exist, generation/scoring run deterministically, non-trivial across seeds, scorer accepts reference and rejects perturbations, semantic review passes, difficulty is in useful range, novelty check passes. Key detail: environment-as-code — unlimited fresh instances, deterministic verifier. Most relevant to Phase F scientific domain environments.

HyperEyes (arXiv:2605.07177) — retrieval/search tool world

Environment = retrieval interface + efficiency-aware verifier. Not Docker-heavy. World pressure from mosaic augmentation, Wikidata random walks, progressive rejection sampling, tight turn budgets, and TRACE efficiency reward. Key detail: the "world" is the action space design + verifier, not a runtime.

RepoLaunch / Harbor — containerized task world

The actual Docker-heavy path: repo → dependency resolution → build → test extraction → runnable SWE environment. Harbor adds the harness: dataset + agent + model + environment provider → parallel rollouts. Key detail: Docker solves resettable isolated state for arbitrary code execution. It is not needed when the environment is a world set or a Python class.

AppWorld — full app-state world

AppWorld is the clearest whole-environment precedent for this roadmap. Environment = simulated users + app databases + app APIs + task generators + state-based evaluators + logs + MCP/HTTP serving. It is not just a benchmark and not just tool schemas. It owns resettable app state, task-specific database diffs, API behavior, ground-truth evaluation, checkpointing, and environment logs. Key detail: a useful whole environment can expose an MCP facade, but MCP is only the interface. The world is the maintained state, API semantics, task generator, and verifier behind it.

What this means for Datalox

Datalox domain env  = stateful tool world (file-backed workspace + MCP tool surface)
Datalox replay      = evidence / logging / replay layer
Datalox fixtures    = declared task worlds (world set = the environment, not one MCP repo)
future Docker       = packaging / reset / isolation layer for arbitrary code execution only

The practical distinction is:

Replay world set
  = finite declared tool world (world packs + task/verifier specs)
  = best for SFT data, preference data, regression, deterministic eval, prompt/model comparison

Executable environment artifact (Phase F)
  = generator + renderer + scorer + verifier metadata
  = EvoEnv-style: generate(seed, difficulty) → unlimited fresh instances
  = Agent-World-style: stateful database + many tools mined from domain literature
  = best for scalable task generation and later online RL

Datalox
  = evidence, replay, registry, fixture, and export layer for both

AppWorld changes the priority of Phase G, but not the immediate wedge. It proves that whole environments need state ownership, reset, task generation, API semantics, and state-based evaluation. Datalox should first record and export rollouts from systems like AppWorld, then use those lessons to define its own scientific reset/step worlds.

Whole Environment Target

A whole Agent RL environment is not just a replay bundle, world pack, Docker image, or benchmark dataset. It is a reset/step contract that owns the task world: 

reset(task_id, seed) -> initial_observation
step(action) -> {
  observation,
  reward,
  done,
  info
}

To qualify as a whole environment, the system must own or explicitly delegate each part of that loop:

Capability Required responsibility Datalox path
Hidden state Initialize and update real task state that the agent cannot directly inspect. Declared in environment specs; stored only as metadata/evidence unless Datalox owns the simulator.
Observation Render the model-visible view after reset and after each action. Record exact agent-visible observations as tool_io_record.v1 when they cross tool boundaries.
Action validation Reject invalid tool names, invalid arguments, impossible operations, and forbidden actions. Reuse tool schemas, fixture manifests, and future action specs; replay mode still misses instead of inventing behavior.
State transition Apply valid actions to the underlying world and produce the next state. Delegate first to runtime adapters such as Docker, WebHarbor-style browser worlds, CyberGym, E2B, Modal, OpenClaw, or custom simulators.
Terminal logic Decide when a task is over and why; the model's "I am done" is only an action. Store terminal reasons and verifier provenance; do not let replay bundles silently define success.
Reward/verifier Score outcome, process, constraints, cost, and anti-hacking checks from environment evidence. Store reward source, version, command, result, and evidence refs; do not make reward logic implicit.

Current fixture replay is a replay-backed API world slice of an environment. It can reproduce observed tool/API behavior. A whole environment must also handle new actions through a real runtime or simulator. If the agent takes an unseen action, replay should miss clearly; a whole environment should step the world.

Runtime Adapter Contract

A runtime adapter is a thin lifecycle and evidence bridge. It should be boring, strict, and unable to invent observations. 

export interface RuntimeAdapter {
  readonly id: string;
  readonly kind: "docker" | "cybergym" | "e2b" | "modal" | "openclaw" | "custom";

  prepare(spec: RuntimeSpec): Promise<PreparedRuntime>;
  start(prepared: PreparedRuntime): Promise<RunningRuntime>;
  exposeTools(runtime: RunningRuntime): Promise<ToolSurface[]>;
  attachRecorder(runtime: RunningRuntime, recorder: ToolIoRecorder): Promise<RecorderHandle>;
  stop(runtime: RunningRuntime): Promise<RuntimeStopResult>;
  collectEvidence(runtime: RunningRuntime): Promise<RuntimeEvidence>;
  invokeVerifier(runtime: RunningRuntime, verifier: VerifierSpec): Promise<VerifierResult>;
}

Important rules:

  • prepare and start may call the external runtime.
  • attachRecorder must capture agent-visible requests and observations as tool_io_record.v1.
  • exposeTools must record the tool catalog or command surface visible to the agent.
  • invokeVerifier may call an external verifier, but Datalox stores only evidence, verifier version, and result metadata.
  • Pure replay mode must not call prepare, start, invokeVerifier, or any live upstream service.
  • Runtime-backed work must use record, rollout, or verify commands that make live execution explicit.

Spec Files

Runtime adapters should be driven by specs, not hand-coded one-off scripts. These specs can live in world packs or private project directories.

Spec Owns Does Not Own
runtime.spec.json Runtime kind, image/env id, startup command, tool endpoint refs, network policy refs. Tool observations or rewards.
task.spec.json Initial user goal, allowed tools, constraints, termination conditions, task family. Runtime lifecycle mechanics.
verifier.spec.json Verifier command/API, version, expected artifacts, rubric refs, anti-hacking checks. Agent scaffold or replay lookup.
scaffold.spec.json Agent harness, prompt version, context manager, tool schema adapter, model-visible rendering. Runtime state or tool payload storage.
fixture-set.json Composition of world packs and optional runtime specs into one declared world. Open-ended behavior for unseen calls.
environment.spec.json Environment kind, generator, renderer, scorer, admission checks, and replay evidence refs. Datalox-owned sandbox implementation or hidden reward logic.

Command Shape

Near-term fixture replay stays runtime-free:

datalox fixtures install ../datalox-api-gym-worlds/fixtures/github-pr-review-basic
datalox fixtures list
datalox replay --fixture github-pr-review-basic@2026-05.0

Later runtime-backed record mode:

datalox runtime record \
  --runtime docker \
  --runtime-spec runtime.spec.json \
  --task task.spec.json \
  --bundle-id support-api-docker-001

Later runtime-backed verification:

datalox runtime verify \
  --bundle .datalox/replay-bundles/support-api-docker-001 \
  --verifier verifier.spec.json

Later runtime-backed rollout for a API world that declares executable context:

datalox runtime rollout \
  --fixture cybergym-cve-task@2026-05.0 \
  --runtime cybergym \
  --runtime-spec runtime.spec.json

This command must mean "activate the declared external runtime adapter and capture tool evidence from a live rollout." It must not mean "Datalox now owns CyberGym's environment builder, vulnerability verifier, container images, or network firewall."

These modes should stay mechanically distinct so users and agents can tell whether a command is serving recorded observations or starting a live executable environment.

datalox runtime rollout Yes Yes Yes, from the rollout No

Domain Tool Servers vs. World Sets

A scientific agent environment is not one MCP server. A real flow cytometry task world may involve the flow cytometry MCP, a filesystem tool, a search tool, a LIMS API, and a report generator. 

domain environment  =  world set  (declared, versioned tool world)
                   ≠  one MCP repo  (that is one tool server / one recorded slice)

The domain MCP repos (datalox-flow-cyto-mcp, datalox-molecule-biology, datalox-protein-mcp) are tool servers. They own one domain's state and tool surface. The world set composes them into the actual environment: 

flowcyto-gating-qc-basic@2026-06.0     // world set = the environment
  fixtures:
    flowcyto-gating-qc-basic@2026-06.0 // recorded flowcyto MCP tool calls
    search-policy-corpus-basic@2026-06.0 // optional recorded search tool calls
    filesystem-ops-basic@2026-06.0     // optional recorded filesystem/report calls
  specs:
    task.spec.json                      // what the agent is asked to do
    verifier.spec.json                  // how success is measured

In replay mode none of the tool servers need to be running. A live tool server is only needed when creating new world packs (rollout / record mode). This is also why Docker alone is not enough: Docker packages runtime deps, but the world set manifest is what declares which tools exist, which versions, what data, what verifier, what reward, and what replay evidence.

Authoring Path vs. Consumption Path

These two workflows are mechanically separate. The training/eval consumer only needs the consumption path — no live tool servers required. 

Authoring path  (rollout owner — needs live tool servers)
live domain MCP / API / CLI
  → Datalox proxy record
  → tool_io_record.v1  (request + observation, keyed by hash + sequence_index)
  → replay_bundle.v1   (sealed, checksummed)
  → world pack       (versioned, published to datalox-api-gym-worlds)
  → world set        (packs + task.spec.json + verifier.spec.json + reset contract)

Consumption path  (training team — no live servers needed)
world set install
  → replay runtime     (serves recorded observations; structured miss on unseen calls)
  → agent rollout trace + verifier result
  → sft_frame.v1 + completion_frame.v1 + preference_pair.v1
  → SFT / preference training / reward modeling / eval / regression
  → online RL later, only with runtime-backed reset/step

Steps 2a–2d are authoring work: they require standing up live tool servers and recording real rollouts. Step 3 and Phase D onwards are consumption-side infrastructure once packs exist.

Training Consumer Order

The roadmap should optimize for the practical adoption order. Agent RL remains a downstream consumer, but it should not be the first promise.

OrderConsumerDatalox outputBoundary
1 SFT-first teams sft_frame.v1 and completion_frame.v1 from successful and recovery trajectories No blind cloning of failed trajectories.
2 Preference training preference_pair.v1: success > recovery > failure under the same task Pairs must point back to replay evidence and task ids.
3 Reward modeling and eval reward_record.v1, verifier provenance, reference reward schemes Datalox records reward evidence; training teams own final reward design.
4 Online RL transition_frame.v1 and runtime-backed rollout evidence Requires reset/step worlds, rollout throughput, and verifier capacity.

Phases

Phase A: Runtime-Free Fixture Replay

Implemented locallyEngine repo

  • Install existing world packs from datalox-api-gym-worlds/.
  • Verify bundled replay artifacts before serving.
  • Replay with upstream off.
  • Return explicit replay misses for unseen calls.

Pass criteria:

  • One existing world pack installs from a local directory.
  • datalox replay --fixture starts a pinned pure replay runtime.
  • Replay startup prints fixture ref, bundle id, checksum, export/redaction status, and tool catalog paths.
  • Missing installed fixtures fail with an agent-readable install-first error.
  • Tampered bundles fail before the replay server starts.
  • No Docker or external sandbox is required.
  • No RuntimeAdapter is instantiated by pure replay.
  • Unseen calls return structured replay_miss payloads with liveFallback: false.

Phase B: World Sets

Implemented locallyEngine repo first

  • Compose multiple world packs into one declared tool world.
  • Reject tool-name collisions unless explicitly namespaced.
  • Preserve per-pack provenance and bundle verification.

Pass criteria:

  • datalox replay --fixtures composes GitHub, Slack, and search fixtures.
  • datalox replay --fixture-set resolves installed fixture-set manifests and member fixture refs.
  • Duplicate tool names across member fixtures fail before serving.
  • Trust output preserves per-fixture bundle ids, checksums, and tool catalog paths.
  • Miss behavior remains deterministic and agent-readable.

Phase C: Spec Metadata Hooks

Implemented locallyEngine and API world repos

  • Add optional task.spec.json, verifier.spec.json, and scaffold.spec.json.
  • Validate specs strictly, but allow fixtures without runtime specs.
  • Preserve spec refs through fixture install, fixture-set install, and replay startup trust output.

Pass criteria:

  • Specs are references and metadata, not hidden executable behavior.
  • Fixture and fixture-set manifests can reference zero or more spec files by safe relative path.
  • Spec ids, versions, refs, task ids, verifier ids, and scaffold ids are deterministic and schema-validated.
  • Install and replay preserve spec metadata in the local fixture cache without executing verifier or runtime code.
  • A fixture-set can declare which task specs are valid for its composed world.
  • The API world repo validates example specs and publishes them into catalog.json.

Step 2: FlowCyto Environment Pack Proof

Implemented locallyEngine + API world repos

Makes the "scientific environments plus replay/data layer" claim operationally true. The target is flowcyto-gating-qc-basic@2026-06.0, defined in flowcyto-environment-pack-plan.html.

  • 2a. In datalox-flow-cyto-mcp, define the first three gating/QC tasks with sample refs, accepted channel pairs, validator thresholds, required report fields, and expected failure codes. Expand to 10+ tasks after the thin proof is stable.
  • 2b. Harden the domain tool contract: open_fcs, get_plot_context, upsert_gate, compute_gate_stats, validate_gate_qc, and submit_report return structured observations, revision ids, evidence refs, and agent-readable errors.
  • 2c. Run first live rollouts against flowcyto-mcp, capture tool I/O, and pack verified replay bundles.
  • 2d. Publish flowcyto-gating-qc-basic@2026-06.0 in datalox-api-gym-worlds with task/verifier refs, bundle checksums, and train/dev/test split.
  • 2e. Verify the consumer path: datalox run --fixture-set flowcyto-gating-qc-basic@2026-06.0 and datalox export sft.

Pass criteria:

  • A fresh user can resolve or install the world set.
  • A task requires at least three domain tools to solve.
  • Wrong or incomplete actions produce structured failures, not fake success.
  • Validators emit pass/fail plus evidence refs.
  • run.json includes messages, tool calls, observations, final answer, fixture refs, and export gate.
  • datalox export sft produces valid sft_frame.v1 JSONL.
  • The first world set includes one success task, one stale-revision failure task, and one report-validation failure task. Preference export is deferred until there is enough success/recovery/failure breadth.

Step 3: Training-Ready Trajectory Exports

NextEngine repo

Covers training-ready checklist items 7, 8, 10. SFT and preference training are the first practical consumers. Online RL comes later.

  • 3a. SFT export exists now. Add completion_frame.v1 next for successful and recovery trajectories. Failed trajectories must be labeled for preference/eval use, not blindly cloned.
  • 3b. Preference pair export. Add preference_pair.v1 for success > recovery > failure comparisons under the same task, with replay evidence refs on both chosen and rejected sides.
  • 3c. Reward evidence and reference schemes. Extend verifier.spec.json to declare reward source, reward version, evidence refs, and optional reference components such as outcome, process, constraint, and cost. Datalox ships reference schemes; training teams own final reward functions and weights.
  • 3d. Transition export later. Add transition_frame.v1 only after the completion/preference/reward evidence path is stable: 
    sft_frame.v1          (model-visible input, target response/action, evidence refs)
completion_frame.v1   (task, full trajectory, outcome, verifier evidence)
preference_pair.v1    (chosen trajectory, rejected trajectory, task id, evidence refs)
reward_record.v1      (source, version, vector/scalar when available, evidence refs)
transition_frame.v1   (state_t, action_t, observation_t, reward_t, done_t)
    Transition frames are the per-step shape RL systems need, but PPO/GRPO still require on-policy rollouts unless the training stack supports offline/replay-conditioned updates. Static replay exports are immediately useful for SFT, reward modeling, offline RL, eval, regression, and debugging. For true online GRPO/PPO, Datalox must connect to a live rollout loop (Phase G + runtime adapters).
  • 3e. Export admission gates: block export when verifier evidence, task spec, or required reward evidence is missing. Declare train/dev/test split in world set manifest.

Pass criteria:

  • sft_frame.v1 and completion_frame.v1 export from a verified world set rollout.
  • preference_pair.v1 exports success/recovery/failure comparisons under the same task.
  • reward_record.v1 stores reward source, version, and evidence refs; reference schemes are marked reference-only.
  • Export blocked when evidence is incomplete. Split declared and respected.

Only after local adapter SPISeparate packages allowed

Phase F: Executable Environment Artifacts

After specs and runtime adaptersRegistry path

The scientific world sets from Steps 2–3 are the first concrete instances of this pattern. Phase F formalizes them into versioned environment artifacts with generation, rendering, scoring, and admission.

Two models informed by papers:

  • EvoEnv-style (arXiv:2605.14392): each scientific domain gets a generate(seed, difficulty) that samples fresh task instances — e.g. synthetic FCS files with known ground-truth gates — instead of a fixed pre-recorded corpus. Unlimited training data with a deterministic verifier. Admitted only after deterministic validation, difficulty calibration, and novelty checks pass.
  • Agent-World-style (arXiv:2604.18292): stateful database + many tools mined from domain literature, MCP specs, and tool docs. Tasks synthesized from tool dependency graphs. Verifier scripts and LLM rubrics from domain knowledge.
  • Accept environment artifacts with generator, renderer, scorer, and admission metadata.
  • Version the environment artifact separately from replay bundles produced by runs inside it.
  • Record scorer/verifier version and anti-hacking checks as evidence metadata.
  • Allow synthesized environments only after deterministic validation passes.

Pass criteria:

  • A generated environment can produce multiple task instances with stable scoring metadata.
  • generate(seed, difficulty) is deterministic for the same world version.
  • Each rollout still emits tool_io_record.v1 and replay_bundle.v1 evidence.
  • Failed admission blocks publication into a shared fixture/environment registry.
  • Difficulty calibration and novelty hash are present in admission metadata.

Phase G: Whole Environment Contract

After executable artifactsReset/step world

  • Define environment_world.v1 as a declared reset/step contract over one task family.
  • Support both replay-backed and runtime-backed worlds without mixing their command modes.
  • Require explicit hidden-state ownership, observation renderer, action schema, terminal rules, and reward/verifier refs.
  • Record every rollout from a world into replay evidence and optional trajectory/transition/completion views.

Pass criteria:

  • reset(task_id, seed) is deterministic for the same world version.
  • step(action) validates actions and returns structured observation, reward, done, and info.
  • Unseen actions in snapshot worlds produce clear replay misses; unseen valid actions in runtime-backed worlds advance the runtime.
  • Terminal reasons and reward evidence are versioned and traceable to recorded evidence.
  • Rollouts produce replay bundles that can be inspected independently of the live runtime.

CyberGym Positioning

CyberGym-like systems are full executable benchmark environments. They may own task construction, sandbox execution, vulnerability-specific artifacts, submission servers, verification scripts, and restricted network behavior.

Datalox should integrate at the boundary:

CyberGym runtime -> agent tool actions -> Datalox tool_io_record.v1
Datalox replay_bundle.v1 -> API world -> eval/export rows

That gives users reproducible evidence and replayable tool-side experience without forcing Datalox to distribute huge benchmark data or maintain a cybersecurity runtime.

WebHarbor Positioning

WebHarbor-style systems are browser-world runtimes. They own local website mirrors, Docker packaging, static assets, SQLite reset state, auth flows, visual fidelity, and human review of site behavior. That is valuable runtime infrastructure, but it is not the first Datalox wedge.

Datalox should integrate at the boundary:

WebHarbor mirror -> browser/tool actions -> Datalox tool_io_record.v1
Datalox replay_bundle.v1 -> world set -> task/verifier/export rows

The product rule is strict: Datalox may reference or start a WebHarbor runtime through a future runtime adapter, but plain datalox replay --fixture still serves only verified recorded observations. If an agent takes an action outside the recorded world, replay misses. A runtime-backed rollout may step the WebHarbor world, but that is a different command mode.

Concrete future shape:

fixture-set/
  manifest.json
  runtime.webharbor.json
  tasks/*.json
  verifiers/*.json
  replay-bundle/<bundle-id>/...

This keeps Datalox narrow without making it small. The core claim is not "we clone every website." The core claim is "we make agent-visible worlds versioned, replayable, auditable, and usable for eval and training, including worlds launched by systems like WebHarbor."

AppWorld Positioning

AppWorld-like systems are full app-state task worlds. They own simulated users, base databases, task-specific database state, app APIs, API semantics, task generation, state-based unit-test evaluation, checkpointing, and environment logs.

Datalox should integrate at the boundary first:

AppWorld MCP/API server -> agent tool actions -> Datalox tool_io_record.v1
Datalox replay_bundle.v1 -> world set -> transition/completion exports

Product rule: Datalox should not present AppWorld as a competitor to beat in the near term. It should treat AppWorld as a reference implementation of a whole environment and a near-term adapter target.

Agent-World And Environment Synthesis Positioning

Agent-World-style systems point toward discovering tool ecosystems, synthesizing verifiable tasks, and training across many environment themes. Environment-synthesis systems point toward reusable executable artifacts that can generate, render, and score many fresh instances.

Datalox should sit under that loop:

environment artifact -> agent rollout -> tool_io_record.v1
tool_io_record.v1 -> replay_bundle.v1 -> fixture/environment registry
registry -> deterministic replay -> grounded training/eval exports

This keeps the claim realistic. Datalox does not have to synthesize every environment itself. It has to make environment outputs versioned, replayable, auditable, and usable by eval and training pipelines.

Non-Goals

  • Do not build distributed filesystems, image streaming, or overlay filesystem infrastructure.
  • Do not own Docker image construction for every task family.
  • Do not own CyberGym's verifier, exploit server, firewall, or benchmark data distribution.
  • Do not clone AppWorld before the API-world/replay evidence layer can package and export AppWorld rollouts.
  • Do not own canonical reward design for training teams; provide reward evidence schemas and reference schemes.
  • Do not claim to synthesize environments before fixture replay and spec validation are solid.
  • Do not hide replay misses by falling back to live services.
  • Do not make runtime adapters required for simple MCP/API fixture replay.
  • Do not put --runtime on plain datalox replay; use runtime-backed command names instead.
