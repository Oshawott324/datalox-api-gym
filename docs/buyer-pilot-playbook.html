Datalox Buyer Pilot Playbook

The first buyer motion should not sell "trajectory recording." It should sell a concrete versioned workflow world that lets a training or eval team reproduce real tool-use failures and export evidence-backed data.

Updated June 3, 2026 - Local pilot artifact

One-Line Offer

We reconstruct one real workflow into a versioned, replayable, verifiable agent task world, then export the runs into the customer's training or eval stack.

The buyer should see Datalox as an environment supplier, not as a data format vendor. Replay is the evidence layer underneath the environment.

Best First Buyers

Post-Training Team

They need tool-use failures, reproducible rollouts, and exportable rows for SFT, preference, reward, or RL experiments.

Workflow Automation Team

They own a costly domain workflow and need to know whether agents can actually perform it with tools, state, and validators.

Inference Provider

They need proof that newer models are better on real workflows, not only chat benchmarks or shallow tool-call demos.

We cannot generate enough realistic agent failures. Use real or reconstructed workflow tools with state, validators, and recorded failure cases. Fixture tasks with success, recovery, and failure metadata.

Pilot Shape

  1. Pick one workflow with clear state transitions, tool outputs, and a pass/fail outcome.
  2. Reconstruct the workflow into agent-native tools or wrap an existing MCP/CLI surface without changing the buyer's production app.
  3. Capture 20 to 50 task instances with at least one baseline model failure mode per task family.
  4. Publish a private fixture set with task specs, verifier specs, replay bundles, train/dev/test split metadata, and export commands.
  5. Run their target model stack against the fixture set and export rows in the format their training team wants.

Pass Criteria

  • The customer's agent can run at least one published fixture set locally or in CI.
  • Replay works with upstream tools off and rejects unseen calls through structured replay miss records.
  • Every accepted task has task metadata, verifier metadata, expected tools, forbidden behavior, and split assignment.
  • At least 20 private pilot tasks contain real workflow evidence, not hand-written chat transcripts.
  • The customer can export one training or eval view into their stack without changing Datalox internals.
  • The buyer can name a model behavior they learned from the replayed workflow that they could not see in their existing evals.

What To Ask The Buyer For

  • Five to ten real workflow examples, including successful and failed attempts.
  • The tools, files, APIs, or CLI commands a human or existing automation uses today.
  • The minimum acceptable output and the domain reasons that make an output wrong.
  • Any data restrictions before capture starts. Do not accept sensitive data until the export policy is explicit.
  • The target consumer: eval harness, SFT pipeline, preference data pipeline, or RL environment wrapper.
The pilot should not promise to replace the buyer's application. It should prove that one high-value workflow can become an agent-native task world with reproducible evidence.
