Physical Lab Replay and Simulation Ladder

Physical lab environments sit above sandbox infrastructure. Fast containers and microVMs can reset local processes, but they cannot reset a paid experiment, a lab queue, an external API account, or a robot run. Datalox should package those workflows into API worlds where replay is the high-trust evidence layer and simulation is added only after real observations exist.

Updated June 7, 2026 - Local strategy doc

Updated Core Realization

Environment infra is the floor, not the product. For physical labs, the scarce object is not a sandbox. It is a resettable, verifiable API world built from real provider workflows, real observations, hidden verifier state, and controlled side-effect boundaries.

The env-scaling memo is right that agentic RL needs fast reset, isolation, checkpoint/restore, and high-throughput rollout infrastructure. But that infra mostly resets local compute state. It does not make a real Adaptyv experiment free, roll back a UniteLabs-connected instrument run, or create a Medra API where none is public. Datalox should therefore own the world layer above sandbox infra: state distributions, tool contracts, replay evidence, verifiers, and exports.

Layer Boundary

Layer Examples What It Solves Why It Is Not Enough
Sandbox infra Docker, microVM, E2B, Daytona, DeltaBox, Crab, cluster rollout workers. Fast process/file reset, isolation, checkpoint, parallel rollout throughput. Cannot reset external SaaS state, lab queues, physical instruments, reagent cost, or scientific outcomes.
Datalox API world Adaptyv protein design world, billing-support miniworld, future SiLA/UniteLabs procedural world. Resettable task scenarios, tool schemas, state, replay evidence, hidden verifier state, exportable runs. Needs provider APIs, real observations, domain curation, and verifier design.
Live provider Experiment API, workcell orchestrator, instrument connector, scientific record system. Ground-truth result, real failure modes, production constraints. Slow, expensive, unsafe to fan out, and often not directly resettable.

Three Environment Types

Type What It Means Main Difficulty
Virtual app env Reconstruct the software workflow in a controlled world. State reset, permissions, realistic app behavior, verifiers.
Physical live env Call real lab APIs and wait for real-world state changes. Cost, time, safety, approvals, partial failures, result provenance.
Physical simulated env Expose the same lab-like endpoints, but return replayed observations or model-predicted experimental outcomes instead of running the physical experiment. Coverage, realism, trust, and matching real assay or instrument outcome distributions.

Why Replay Comes First

Real experiments are expensive. They take time, consume reagents and samples, use instrument capacity, and may fail for physical reasons. One live run should therefore become a reusable asset for many agent runs. 

live provider API run
  -> capture exact endpoint observations
  -> capture timestamps and state transitions
  -> capture result files and verifier outcomes
  -> package as replay-backed API world
  -> let many agents run against the same episode cheaply

Replay is not a broad simulator. It is a narrow, high-trust simulator. It returns what actually happened for recorded requests and rejects unseen calls through replay misses.

Updated Simulation Ladder

Layer Strength Weakness Use
L0 Live provider endpoint Real result, real constraints, real failures. Expensive, slow, scarce, risky. Ground truth, selected validation, and new evidence collection.
L1 Replay-backed API world Exact observations from real runs; high trust. Narrow coverage; cannot answer unrecorded physical questions. Regression, evaluation, SFT, failure analysis.
L2 Stateful API world Cheap resettable scenarios over provider-like state machines: drafts, quotes, queues, permissions, failures. Can model workflow state without proving new scientific outcomes. Agent practice, verifier development, safe failure generation, task distribution scaling.
L3 Calibrated outcome simulator Predicts plausible physical experiment outcomes for new proposed actions, such as binding, expression, yield, QC metrics, instrument errors, or run status transitions. Lower trust unless calibrated against real lab runs; can invent outcomes that look plausible but are scientifically wrong. RL scale, branch exploration, stress testing, and deciding which new live experiments are worth paying for.
L4 Live-gated hybrid world Agents practice in replay/sim, then selected high-value actions are approved for real provider execution. Requires budgets, approvals, safety policies, provider partnership, and result ingestion. Closed-loop lab learning where each live run becomes new replay evidence and simulator calibration data.
In this document, "generative simulator" means an outcome model for the physical experiment. It is not just a chatbot roleplaying a lab. Given an action such as "submit these variants" or "run this workcell protocol," it predicts lab-like outputs: assay values, pass/fail states, delays, result files, and recoverable or unrecoverable errors.

Long-Running Lab Loop

The target task is not "compute a protein property" or "gate one FCS sample." Those can be useful subtools. The real task is a full lab lifecycle. 

agent reads target and constraints
  -> designs candidates or selects workflow
  -> estimates cost and asks for approval when needed
  -> submits experiment or workcell run
  -> waits through queued/running/completed states
  -> retrieves structured results and files
  -> updates the scientific record
  -> decides the next action
  -> Datalox records evidence, verifies outcome, and exports rows

Provider Roles

Experiment API

Example: Adaptyv. The agent submits designs, tracks experiment state, receives wet-lab results, and chooses a next round.

Workcell API

Example: Automata, UniteLabs, Biosero. The agent validates workflows, starts runs, monitors devices, handles recoverable errors, and collects outputs.

Record API

Example: Benchling. The agent reads inventory and assay context, writes results, updates workflow state, and keeps the scientific record consistent.

Adaptyv Build Plan

Do not equate task count with live experiment count. Twenty API Gym tasks should mean twenty replay/scenario/eval tasks over a small number of recorded or seeded provider episodes. It must not mean twenty paid wet-lab runs, twenty material orders, or twenty quote confirmations.

Build Adaptyv in stages so Datalox proves the world contract before spending real lab budget. The first version should be a replay-backed API world with no live mutation path. Live execution comes later behind an explicit approval gate.

Stage What To Build Live Cost Exposure Pass Criteria
V0 offline replay proof Create adaptyv-foundry-basic@2026-06.0 from public docs and curated replay records: target catalog, target detail, cost estimate, draft experiment, experiment status, quote view, and seeded result retrieval. None. No Adaptyv token required; no provider call required. Install, replay, run, verifier metadata, and SFT/eval export work locally.
V1 safe live recording Record only safe endpoints: target browse, target detail, cost estimate, draft creation if approved, experiment read, quote read, and historical result read. Near zero if restricted to read/draft/cost-estimate paths. Quote confirmation stays impossible. Every live call becomes tool_io_record.v1; every replayed call has deterministic request hash and replay miss behavior.
V2 task set Create 20 task scenarios over the same small evidence pool: target selection, cost-aware design, draft setup, status triage, result interpretation, next-round planning, safety-boundary refusal, replay-miss recovery. Still near zero. More tasks reuse the same replay evidence and seeded state branches. Train/dev/test split, task specs, verifier specs, scaffold spec, expected behavior, and eval prompts validate.
V3 live-gated episodes Add an explicit live gate for rare actions such as quote confirmation or paid experiment submission. Controlled by budget, human approval, provider account policy, and task allowlist. No paid call can happen from replay/eval mode. Live mode logs approval id, cost estimate, quote id, and final result provenance.

Adaptyv Task Design Rules

Rule Concrete Meaning
Tasks are scenarios, not experiments A task can ask the agent to inspect replayed target data, estimate cost, compare candidate sequences, or explain a result. It does not require a new physical experiment unless it is explicitly a live-gated task.
Use one episode many ways One recorded provider episode can generate multiple tasks: budget check, safety refusal, result interpretation, status recovery, and next-round planning.
Separate workflow truth from scientific truth Draft, quote, status, and permission behavior can be deterministic. New binding, expression, or yield outcomes must come from replayed results or a calibrated simulator, never from casual LLM invention.
Gate paid execution confirm_quote, material ordering, and any lab-starting action must be absent in V0/V1 replay worlds or return a live-gate-required result.
Track marginal cost Every world and task set should report live experiments used, replay episodes used, estimated live cost avoided, and whether any task requires paid execution.

Datalox Boundary

Datalox Owns Datalox Does Not Own
API world package format, state/task distributions, tool contract, task specs, verifier specs, replay evidence, replay miss behavior, live-gate policy metadata, export adapters. The physical lab, sandbox engine, trainer, Kubernetes, GPU cluster, robot control algorithms, low-level provider SDKs, or customer scientific policy.
Do not sell a replay file as the product. Sell the ability to turn each expensive physical episode into a runnable, verifiable, replayable agent API world. Run it on existing sandbox infra; do not become the sandbox infra.

Business Wedge

A lab team should care because every live experiment can be reused many times for model evaluation, regression testing, and training data export. The same captured episode can test many agents without paying the lab cost again. Infra teams should care because Datalox gives their rollout workers realistic external worlds instead of only local file-system sandboxes. 

real lab episode
  -> high-trust replay-backed API world
  -> many cheap agent rollouts
  -> identify missing branches
  -> run selected new live experiments
  -> optionally train an outcome simulator for broader branches
  -> validate important policies back on the live lab

The long-term shape is hybrid: replay for trust, outcome simulation for scale, live lab execution for validation and new data, and sandbox infra underneath for cheap parallel execution.
