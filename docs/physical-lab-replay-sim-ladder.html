Physical Lab Replay and Simulation Ladder

Physical lab environments are different from virtual app environments. The live lab is expensive and slow, so replay is the first simulator: exact observations from real runs become reusable agent worlds.

Updated June 4, 2026 - Local strategy doc

The Core Realization

For a physical lab, the environment is not only software state. It is an endpoint connected to real-world execution, plus recorded or predicted experiment outcomes for agent training and evaluation.

In a virtual app environment, Datalox may need to reconstruct the app: workspace state, tools, permissions, reset, and verifier logic. In a physical lab environment, Datalox should first wrap existing providers such as experiment platforms, workcell orchestrators, instrument integration layers, and scientific record systems.

Three Environment Types

Type What It Means Main Difficulty
Virtual app env Reconstruct the software workflow in a controlled world. State reset, permissions, realistic app behavior, verifiers.
Physical live env Call real lab APIs and wait for real-world state changes. Cost, time, safety, approvals, partial failures, result provenance.
Physical simulated env Expose the same lab-like endpoints, but return replayed observations or model-predicted experimental outcomes instead of running the physical experiment. Coverage, realism, trust, and matching real assay or instrument outcome distributions.

Why Replay Comes First

Real experiments are expensive. They take time, consume reagents and samples, use instrument capacity, and may fail for physical reasons. One live run should therefore become a reusable asset for many agent runs. 

live provider API run
  -> capture exact endpoint observations
  -> capture timestamps and state transitions
  -> capture result files and verifier outcomes
  -> package as replay world
  -> let many agents run against the same episode cheaply

Replay is not a broad simulator. It is a narrow, high-trust simulator. It returns what actually happened for recorded requests and rejects unseen calls through replay misses.

Simulation Layers

Layer Strength Weakness Use
Live lab Real result, real constraints, real failures. Expensive, slow, scarce, risky. Validate important policies and collect new evidence.
Replay simulator Exact observations from real runs; high trust. Narrow coverage; cannot answer unrecorded physical questions. Regression, evaluation, SFT, failure analysis.
Generative simulator Predicts plausible physical experiment outcomes for new proposed actions, such as binding, expression, yield, QC metrics, instrument errors, or run status transitions. Lower trust unless calibrated against real lab runs; can invent outcomes that look plausible but are scientifically wrong. RL scale, branch exploration, stress testing, and deciding which new live experiments are worth paying for.
In this document, "generative simulator" means an outcome model for the physical experiment. It is not just a chatbot roleplaying a lab. Given an action such as "submit these variants" or "run this workcell protocol," it predicts lab-like outputs: assay values, pass/fail states, delays, result files, and recoverable or unrecoverable errors.

Long-Running Lab Loop

The target task is not "compute a protein property" or "gate one FCS sample." Those can be useful subtools. The real task is a full lab lifecycle. 

agent reads target and constraints
  -> designs candidates or selects workflow
  -> estimates cost and asks for approval when needed
  -> submits experiment or workcell run
  -> waits through queued/running/completed states
  -> retrieves structured results and files
  -> updates the scientific record
  -> decides the next action
  -> Datalox records evidence, verifies outcome, and exports rows

Provider Roles

Experiment API

Example: Adaptyv. The agent submits designs, tracks experiment state, receives wet-lab results, and chooses a next round.

Workcell API

Example: Automata, UniteLabs, Biosero. The agent validates workflows, starts runs, monitors devices, handles recoverable errors, and collects outputs.

Record API

Example: Benchling. The agent reads inventory and assay context, writes results, updates workflow state, and keeps the scientific record consistent.

Datalox Boundary

Datalox Owns Datalox Does Not Own
World package format, tool contract, task specs, verifier specs, replay evidence, replay miss behavior, export adapters. The physical lab, sandbox engine, trainer, Kubernetes, GPU cluster, robot control algorithms, or customer scientific policy.
Do not sell a replay file as the product. Sell the ability to turn each expensive physical episode into a runnable, verifiable, replayable agent world.

Business Wedge

A lab team should care because every live experiment can be reused many times for model evaluation, regression testing, and training data export. The same captured episode can test many agents without paying the lab cost again. 

real lab episode
  -> high-trust replay world
  -> many cheap agent rollouts
  -> identify missing branches
  -> run selected new live experiments
  -> optionally train an outcome simulator for broader branches
  -> validate important policies back on the live lab

The long-term shape is hybrid: replay for trust, outcome simulation for scale, and live lab execution for validation and new data.
