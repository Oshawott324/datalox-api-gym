Budget Agent RL Proof Demo Guide

A concrete traction-oriented plan for showing that Datalox can turn versioned agent environments into low-cost, measurable model improvement. The goal is not only to prove the architecture. The goal is to make the product feel useful: bring a small model, run a declared environment, train cheaply, and show a better held-out result with replayable evidence.

Updated May 27, 2026 - Local planning doc

Demo Claim

Datalox can produce a small but real agent-training loop under a constrained budget: a small open model improves on held-out tool-use tasks after SFT, preference training, and a tiny verifier-backed RL run, with every result tied back to replay evidence. 
versioned fixture set
  -> baseline run
  -> grounded SFT / preference / reward exports
  -> low-cost LoRA training
  -> held-out eval
  -> replayable before/after evidence

Public wording should be disciplined: "budget agent training with replayable evidence," not "we built frontier agent RL."

Recommended Scope

Do not start the budget RL traction demo with FlowCyto. Flow cytometry is the right serious second proof, but it adds domain expertise, plot interpretation, and possibly vision-model cost before the training loop itself is proven. The first public demo should be text-first, tool/API based, deterministic, and cheap enough to rerun several times.

Also do not treat the existing support-triage-basic fixture as the public traction dataset unless it is rebuilt from real observed cases. The current support fixture is good for plumbing and schema smoke tests. The public demo needs tasks sourced from real incidents, issues, support threads, CI logs, or design-partner traces where baseline agents actually fail.

ChoiceUseReason
real-ci-triage-basic@2026-06.0 First public budget RL traction demo Real open-source PR/issue/check-run artifacts are public, text-first, cheap to replay, and agents often fail by missing the key log, overclaiming, or proposing the wrong next action.
support-triage-real-basic@2026-06.0 Design-partner support demo Best buyer story if sourced from anonymized real support traces, but avoid it for public launch unless provenance and redaction are clean.
support-triage-basic@2026-06.0 Pipeline smoke only Useful for exercising fixture composition, exports, and reward schemas. Too synthetic and too broad for the public traction claim.
flowcyto-gating-qc-basic@2026-06.0 Second serious proof Best product story for scientific environments, but too risky for the first budget training claim because success may depend on domain reasoning and visual/plot interpretation.
bio-agent-basic@2026-06.0 Third breadth proof Shows environment-family breadth after the text-first budget RL loop and FlowCyto proof both work.
First demo selection rule: no vision dependency, no specialist scientific interpretation, no long-horizon UI loop, and no verifier that needs a judge model. The first win should come from tool-use behavior that a small text model can realistically improve.

The first task family should look like: 

read real issue / PR / check-run / incident thread
  -> retrieve logs, docs, policy, or prior discussion
  -> identify the failing condition or missing evidence
  -> choose next action: fix, ask, escalate, retry, or close as duplicate
  -> cite exact evidence
  -> emit structured triage package

Paper-Backed Task Prior

Use recent agent benchmark papers to choose the task family. Do not spend weeks discovering difficulty from scratch. The papers are a prior: they identify failure modes that are already hard for strong agents. Datalox should still admit only tasks with real source artifacts and at least one observed baseline failure.

SourceHardness SignalDatalox Demo UseFirst-Demo Decision
Terminal-Bench 2.0 Hard terminal workflows with human solutions and test-based verification; frontier agents reported under 65%. Use as the strongest prior for CI/log/package/debug tasks because verification can be deterministic and cheap. Primary source family.
TerminalWorld Tasks reverse-engineered from in-the-wild terminal recordings; verified subset still hard for current agents. Use the collection method as a model: real workflows first, validated task after, not synthetic prompt first. Primary sourcing pattern.
SWE-bench / SWE-bench Verified / SWE-bench Live Real GitHub issues exposed underspecification, weak tests, contamination, and environment setup limits. Use recent public issues, PRs, failing checks, and maintainer discussion as source artifacts. Avoid claiming leaderboard comparability. Good source pool, but use fresh/recent artifacts where possible.
tau-bench Policy-constrained tool agents fail on multi-turn support tasks even when final state is objectively checkable. Use for the support task shape: policy + API state + customer request + forbidden actions + final database state. Good private design-partner path; not first public path unless traces are real and shareable.
BrowseComp Hard-to-find but easy-to-verify evidence retrieval is a clean benchmark design pattern. Use the asymmetry: make tasks where finding the key evidence is hard, but the verifier checks a short answer plus cited source ids. Useful subtask pattern, not a full environment by itself.
WebArena / AppWorld / OSWorld Realistic web, app, and desktop agents struggle with long horizons and stateful environments. Use for roadmap validation, not the first budget demo, because browser/desktop/vision loops raise cost and flakiness. Defer.
MLE-bench / PaperBench End-to-end ML engineering and paper replication are clearly hard, but expensive and long-running. Use as evidence that agentic training matters; do not use for the first budget claim. Exclude from first demo.
Practical rule: papers choose the hunting ground; replay evidence chooses the final dataset. We can skip broad exploration, but not the one cheap baseline pass that proves the selected task is actually failing in our harness.

TerminalWorld Domain Fit

TerminalWorld is close to Datalox's domain because it starts from real workflow recordings and converts them into reproducible, verified tasks. It is not a FlowCyto paper, but it validates the same product thesis: hard agent tasks should come from observed work, not invented prompts.

TerminalWorld PieceDatalox InterpretationDomain Extension
Public terminal recordings Source traces for fixture authoring, analogous to live MCP/API/tool rollouts. Mine real CLI, notebook, pipeline, or support workflows before writing task specs.
Intent inference and oracle extraction Candidate task spec plus success trajectory, downstream from evidence. Use agents to draft task/verifier specs, then require replay evidence and human review.
Docker execution environment Layer 1/2 runtime owned by benchmark or domain environment, not by Agent Replay core. Datalox records and replays the agent-visible tool I/O around that runtime.
State-based tests Cheap deterministic verifier, suitable for budget RL reward records. Prefer file/state/JSON/test assertions over judge-model scoring.
Verified subset Admission gate: clarity, artifact completeness, oracle replayability, verifier validity. Use the same gate for public fixture sets before claiming traction.

The directly related scientific wedge is not generic terminal use. It is scientific terminal work that can be verified cheaply: 

real bioinformatics / analysis workflow
  -> captured command, file, API, or notebook observations
  -> deterministic artifact or state verifier
  -> Datalox replay bundle
  -> SFT / preference / reward export
Domain CandidateWhy It FitsWhy It Is Cheaper Than FlowCyto Vision
Bioinformatics pipeline triage Real Nextflow/Snakemake/FASTQ/QC failures produce logs, configs, and file outputs. Text/log/file verification; no image interpretation required.
Scientific data cleaning CSV, FASTA, metadata, and notebook regeneration tasks have concrete output artifacts. Verifier can compare schema, rows, checksums, plots metadata, or test outputs.
Flow cytometry script/debug path FCS metadata, marker/channel mismatch, compensation config, and gating script errors are real domain work. Starts with logs/metadata/state diffs before adding plot or vision-heavy judgement.
So the answer is yes: TerminalWorld is related to our domain if we define the first Datalox scientific wedge as evidence-backed workflow replay and deterministic verification. It is not enough to say "terminal benchmark"; the useful adaptation is "TerminalWorld for scientific workflows, with Datalox replay bundles as the evidence layer."

Real Task Bar

A task should enter the budget RL demo only if it passes all of these gates:

GateRequirementReason
Real source artifact Public GitHub issue/PR/check logs, public incident report, public docs, or approved anonymized support trace. A buyer should believe the task represents real work, not a toy puzzle.
Observed agent failure At least one baseline agent run fails through missing evidence, wrong tool path, hallucinated state, or unsupported action. The demo should train on problems agents actually get stuck on.
Deterministic verifier Verifier checks exact evidence ids, expected diagnosis, required missing fields, and forbidden claims. Budget RL needs cheap rewards; judge-model scoring defeats the point.
Replayable observations All issue, log, doc, search, and status observations are captured as tool I/O records. Datalox's value is replayable evidence, not screenshots of a one-off demo.
Teachable difficulty Baseline failure rate should be high enough to improve, but not so high that the task requires hidden expertise. A small LoRA run needs dense, learnable signal.

Specific Real Task Sources

These are concrete source pools for real-ci-triage-basic. Each task should start from a real artifact, then be captured into replay bundles. The fixture should not contain invented tickets.

The first public set should include actual links or provenance ids for every source artifact. If the artifact cannot be shared, keep it for a private design-partner pilot and use public GitHub/incident tasks for the public demo.

Where Initial SFT Data Comes From

The papers and real source artifacts do not directly become SFT rows. They only define the task world. Initial SFT data comes from successful or repaired trajectories on the training split, after those trajectories pass the deterministic verifier. 

paper-backed task family
  -> real source artifact: issue, PR, CI log, support trace, scientific workflow
  -> task.spec.json + replay_bundle.v1 + verifier.spec.json
  -> train/dev/test split by task id
  -> solve only train tasks with human, teacher model, or repaired agent trace
  -> keep trajectories that pass verifier
  -> export prompt/response/tool-step rows as sft_frame.v1
SFT SourceHow It Is MadeUseRisk Control
Human reference trace A human reads the replayed evidence and writes the correct structured answer, optionally with the intended tool path. Highest-quality seed data for the first 20-50 train tasks. Must pass the same verifier; never write references for test tasks.
Teacher-agent success trace A stronger model runs against the replay fixture, uses the tools, and produces a passing answer. Cheap way to scale from seed tasks to 80-200 train tasks. Keep only verifier-passing traces with exact evidence refs; discard unsupported claims.
Repaired baseline trace A weak model fails, then a human or teacher corrects the missing evidence, wrong next action, or final package. Good recovery data because it teaches common failure corrections. SFT target is the repaired success path, not the original failed answer.

A single task can yield multiple SFT rows: one row for choosing the right tool/query, one row for reading the returned evidence, and one row for the final structured triage package. Failed baseline runs are useful for preference pairs and reward records, but should not be blindly imitated by SFT.

First practical batch: pick 30 train tasks, write or teacher-generate 1 verified success trace per task, export those traces to SFT, then use failed baseline traces as negative examples for preference and RL.

Reward Design

Keep reward simple, deterministic, and decomposed. Datalox should preserve reward evidence and reference schemes, not claim to own every customer's final reward function.

Reward ComponentShapeEvidence
Outcome 0 or 1 Verifier pass/fail and terminal reason.
Tool contract 0 to 1 No replay misses, valid tool names, valid arguments, no impossible or irrelevant tool path.
Evidence completeness 0 to 1 Required policy citations, ticket facts, source refs, or domain evidence records.
Efficiency 0 to 1 Tool rounds, duplicate calls, latency, and cost metadata when available.
Report quality 0 to 1 Structured response validator, not free-form final prose alone. 
reference_reward =
  0.50 * outcome
  + 0.20 * tool_contract
  + 0.15 * evidence_completeness
  + 0.10 * efficiency
  + 0.05 * report_quality

The scalar fold is for the demo. The stored artifact should keep the reward vector, source, version, verifier command, and evidence refs.

Training Ladder

StageMethodInputOutputWhy It Is In The Demo
0 Baseline eval Frozen fixture-set test split Pass rate, reward vector, replay miss rate, tool errors Creates the before number and protects against fake progress.
1 SFT LoRA sft_frame.v1 from verifier-passing train-task success and recovery trajectories Cheap first lift Most teams understand SFT and can reproduce it.
2 DPO / preference LoRA preference_pair.v1: success > recovery > failure Tool-use preference correction Shows Datalox can turn replay evidence into preference data.
3 Tiny GRPO / REINFORCE++ Prompt set + deterministic reward function Verifier-backed RL result Completes the "agent RL" story, but stays small.

Concrete Commands

These are target demo commands. Some export and trainer commands may need thin scripts before the demo can be run end to end. The public target is real-ci-triage-basic; the current synthetic support-triage-basic fixture is only a pipeline smoke target.

1. Install The Fixture Set

datalox fixture-sets install real-ci-triage-basic@2026-06.0 \
  --catalog ../datalox-replay-fixtures/catalog.json

2. Serve The Baseline Model

vllm serve Qwen/Qwen2.5-1.5B-Instruct \
  --api-key token \
  --host 127.0.0.1 \
  --port 8000

3. Run The Baseline Eval

datalox eval \
  --fixture-set real-ci-triage-basic@2026-06.0 \
  --catalog ../datalox-replay-fixtures/catalog.json \
  --model Qwen/Qwen2.5-1.5B-Instruct \
  --base-url http://127.0.0.1:8000/v1 \
  --api-key token \
  --split test \
  --out runs/budget-rl/baseline.test.jsonl \
  --json

4. Export Training Views

datalox export sft \
  --runs runs/budget-rl/train \
  --out exports/budget-rl/train.sft.jsonl \
  --json

datalox export preference \
  --runs runs/budget-rl/train \
  --out exports/budget-rl/train.preference.jsonl \
  --json

datalox export rewards \
  --runs runs/budget-rl/train \
  --out exports/budget-rl/train.rewards.jsonl \
  --json

5. Train Small Adapters

python training/sft_lora.py \
  --model Qwen/Qwen2.5-1.5B-Instruct \
  --train exports/budget-rl/train.sft.jsonl \
  --dev exports/budget-rl/dev.sft.jsonl \
  --out models/budget-rl-sft-lora

python training/dpo_lora.py \
  --model models/budget-rl-sft-lora \
  --pairs exports/budget-rl/train.preference.jsonl \
  --dev-pairs exports/budget-rl/dev.preference.jsonl \
  --out models/budget-rl-dpo-lora

6. Run Tiny RL

python training/grpo_budget.py \
  --model models/budget-rl-dpo-lora \
  --tasks exports/budget-rl/train.prompts.jsonl \
  --reward exports/budget-rl/reward_config.json \
  --out models/budget-rl-grpo-lora \
  --max-steps 200 \
  --num-generations 4

7. Re-Evaluate

vllm serve models/budget-rl-grpo-lora \
  --api-key token \
  --host 127.0.0.1 \
  --port 8001

datalox eval \
  --fixture-set real-ci-triage-basic@2026-06.0 \
  --catalog ../datalox-replay-fixtures/catalog.json \
  --model budget-rl-grpo-lora \
  --base-url http://127.0.0.1:8001/v1 \
  --api-key token \
  --split test \
  --out runs/budget-rl/grpo.test.jsonl \
  --json

Outcome Bar

A traction demo needs an outcome a user can understand in one screen.

MetricMinimumGoodGreat
Held-out full pass rate +5 absolute points +10 to +15 points +20+ points
Verifier reward +0.05 average reward +0.10 average reward +0.15+ average reward
Tool contract errors 10% reduction 30% reduction 50% reduction
Replay misses No increase Reduction Near zero on known task paths

If full pass rate does not move, a sub-reward lift can still be useful for internal debugging, but it is not enough for the public traction demo.

Demo Script

  1. Show the fixture set: task specs, verifier specs, replay bundles, train/dev/test split.
  2. Run the small baseline model against the held-out split.
  3. Show failures with replay evidence: missed log evidence, wrong next action, unsupported live mutation, missing fields, or weak report output.
  4. Export SFT, preference, and reward rows from verified evidence.
  5. Show the training command and live cost log.
  6. Run the tuned model on the same held-out split.
  7. Show before/after metrics and open one replay bundle from each side.
  8. End with the product message: budget training that is reproducible, inspectable, and environment-grounded.

Risks And Controls

RiskControl
Eval leakage Freeze test split before training. Store split refs in fixture-set manifest.
Reward hacking Use hidden verifier state and evidence refs. Do not reward final prose alone.
Training on failures as imitation SFT only on success/recovery rows. Failures go to preference/eval, not blind imitation.
Overclaiming Say "improved one small model in one versioned environment," not "solved agent RL."
Cost creep Hard budget gates: $150 smoke, $500 internal proof, $1,500 public demo.
Pipeline fragility Keep first demo LoRA-only. No reward model. No full-parameter training.
