FlowCyto Environment Pack Plan

Concrete plan for proving Datalox's environment wedge: ship one versioned scientific agent environment pack where trajectories come from executable domain tools, workspace state, validators, and replay evidence.

Updated May 25, 2026 - Local planning doc

Target Artifact

Build flowcyto-gating-qc-basic@2026-06.0: a versioned flow cytometry task world where an agent gates cell populations, computes QC statistics, passes validators, and exports replay-backed SFT data.

This should prove that Datalox is not just recording trajectories. The environment must produce trajectories that are executable, replayable, and checkable.

User-Facing Demo

vllm serve Qwen/Qwen2.5-7B-Instruct --api-key token

datalox run \
  --fixture-set flowcyto-gating-qc-basic@2026-06.0 \
  --base-url http://127.0.0.1:8000/v1 \
  --model Qwen/Qwen2.5-7B-Instruct \
  --api-key token \
  --prompt "Gate the lymphocyte population and produce QC stats." \
  --out runs/flowcyto-qwen \
  --json

datalox export sft \
  --run runs/flowcyto-qwen \
  --out exports/flowcyto-qwen.sft.jsonl \
  --json

The first demo should work with a local OpenAI-compatible model endpoint. Datalox should not require users to run an RL trainer to see value.

Agent Tool Path

open_fcs
  -> get_plot_context
  -> upsert_gate
  -> compute_gate_stats
  -> validate_gate_qc
  -> submit_report

The tool sequence is not hard-coded. This is the expected successful path. submit_report validates and persists the report content written by the agent; it does not generate prose. The environment should also expose structured errors for invalid gates, stale workspace revisions, missing channels, missing evidence records, and missing report fields.

Why This Is Hard To Replace With Synthetic Trajectories

Stateful Workspace

Gate writes mutate flowcyto.workspace.json and increment revisions. Later tool observations depend on earlier actions.

Domain Validation

Validators check geometry, channel selection, population percentage, marker stats, and required QC outputs.

Replay Evidence

Every tool call and observation can be captured as replay evidence and exported only after the run is grounded.

Implementation Phases

  1. Phase 1 - Domain task spec. Start with three deterministic tasks: one success path, one stale-revision failure path, and one report-validation failure path. Expand to 10+ tasks only after the thin proof is stable.
  2. Phase 2 - Tool contract hardening. Ensure each MCP tool returns agent-first structured observations, revision ids, error codes, and next-action hints when recovery is clear.
  3. Phase 3 - Validators. Add deterministic checks for gate existence, polygon validity, channel match, population percent range, stats evidence records, QC evidence records, and report fields.
  4. Phase 4 - Live rollout capture. Run an agent through the live flowcyto MCP, capture tool I/O records, and pack verified replay bundles.
  5. Phase 5 - Fixture publication. Publish flowcyto-gating-qc-basic@2026-06.0 in datalox-api-gym-worlds with bundle checksums and train/dev/test split.
  6. Phase 6 - Training export. Run datalox run against the world set and export sft_frame.v1. Add completion and preference exports after success/failure/recovery labels exist.

Pass Criteria

  1. A fresh checkout can install or resolve the world set.
  2. datalox run works against a local OpenAI-compatible endpoint.
  3. The agent must use at least three domain tools to solve a task.
  4. Wrong or incomplete actions produce structured failures, not fake success.
  5. The validator emits pass/fail plus evidence refs.
  6. run.json contains messages, tool calls, observations, final answer, fixture refs, and export gate.
  7. datalox export sft produces valid sft_frame.v1 JSONL.
  8. The first proof contains three deterministic tasks; preference export is not claimed until the world set has enough success/recovery/failure breadth.

What Not To Do

  • Do not make this a generic trajectory-format project.
  • Do not make Datalox own the final reward function.
  • Do not claim online RL before reset/step rollout support exists.
  • Do not rely on model-synthesized tool returns as proof of environment value.
  • Do not overbuild one perfect simulator before proving exportable task breadth.
