Local Engine To Server-Ready Plan

This plan defines what to do in datalox-agent-replay after Option A: keep this repo as the local MCP VCR engine, make it plug-and-play, shape it for a future server registry, and keep curated fixture packs separate.

Status: implementation and cleanup plan. Updated: 2026-05-20. Canonical boundary: project-definition.md, agentic-rl-layer-map.md, replay-bundle-schema.md.

The product sentence should stay stable: Datalox records real MCP/tool behavior once and serves it later as a verified, deterministic replay surface with upstream off.

This repo

datalox-agent-replay is the engine.

It owns proxy record mode, replay mode, bundle packing, bundle verification, local storage, SDK entrypoints, CLI UX, and server-shaped interfaces.

Future repo

datalox-replay-fixtures is the curated fixture library.

It should contain versioned public fixture packs such as GitHub issue workflows, Slack support threads, search/RAG corpora, and SaaS API edge cases.

Future server

The hosted service is a replay bundle registry first.

It stores, verifies, indexes, approves, shares, and serves bundles. It does not need to be the first recorder.

Decision

Do not turn this repo into a large fixture/data repository. This repo should remain the installable engine that can consume local private bundles, checked-in reference bundles, and later remote fixture packs from a registry.

The correct split is: 

datalox-agent-replay   = runtime, CLI, SDK, store, verifier, MCP VCR
datalox-replay-fixtures = versioned public replay fixture packs
Datalox server          = registry, auth, approvals, object storage, sharing, metrics

The engine must not depend on the future fixture repo or hosted server. The fixture repo and server should depend on the engine contract.

Product Boundary

What Datalox should do

  • Record real MCP/tool requests and observations.
  • Snapshot agent-visible MCP tools/list catalogs.
  • Pack verified replay_bundle.v1 artifacts.
  • Serve replay from bundle with upstream completely off.
  • Return deterministic replay misses for unseen requests.
  • Expose local CLI and SDK surfaces.
  • Expose server-shaped storage interfaces and APIs.
  • Measure replay hit rate, miss rate, tool rounds, latency, and cost metadata.

What Datalox should not do here

  • Do not build coding sandboxes or Docker environment rebuilders.
  • Do not build stateful behavioral simulators for unseen actions.
  • Do not write reward functions or judge agents.
  • Do not lead with schema as the main user-facing wedge.
  • Do not put a large curated fixture library in this repo.
  • Do not reintroduce the legacy wiki/note/promotion loop.

The precise wording is important: Datalox can serve recorded observations as record-based mocks. It does not construct open-ended behavioral mocks that respond faithfully to arbitrary unseen actions.

North Star Metric

The immediate product benchmark is Time To First Verified Replay. 

existing MCP server
  -> datalox proxy record
  -> replay bundle
  -> upstream off
  -> deterministic replay
  -> tamper check fails

The local demo target is under 2 minutes with the included fixture. The bring-your-own MCP target is under 5 minutes for a developer who already has an MCP server command.

Implementation Roadmap For This Repo

Phase 1: Verified Replay Quickstart

Build the product proof path as a repeatable local demo. This should become the first public video and the first design-partner smoke test.

Owner: this repoPriority: highest

Add or update:

package.json
examples/verified-replay-quickstart/
  README.md
  fixture-upstream.mjs
  run-demo.mjs
  datalox.record.json
docs/verified-replay-quickstart.md
tests/verifiedReplayQuickstart.test.ts

Target command:

npm run demo:verified-replay

The command must show, in order:

  1. Build the TypeScript project if needed.
  2. Start a deterministic MCP upstream fixture.
  3. Start Datalox proxy in record mode.
  4. Call two or three tools through the proxy.
  5. Pack a replay bundle.
  6. Verify the replay bundle.
  7. Stop upstream.
  8. Start Datalox proxy in replay mode from the bundle.
  9. Replay the same calls successfully.
  10. Call an unrecorded request and show a structured replay miss.
  11. Tamper with a copy of the bundle and show verification failure.

Pass criteria:

  • Demo succeeds on a clean checkout with no secrets and no network dependency.
  • Replay works while upstream is off.
  • Tamper failure is visible and deterministic.
  • Output includes bundle id, tool record count, catalog count, hit count, miss count, and elapsed time.
  • Demo does not mention trajectory rows as the primary value.

Interfaces:

export interface ReplayStore {
  putToolIoRecord(record: ToolIoRecordV1): Promise<void>;
  getToolIoRecord(recordId: string): Promise<ToolIoRecordV1>;
  findReplayRecord(input: ReplayLookupInput): Promise<ReplayLookupResult>;
  putMcpToolCatalog(catalog: McpToolCatalogV1): Promise<void>;
  putReplayBundle(bundle: ReplayBundleV1): Promise<void>;
  verifyReplayBundle(bundleId: string): Promise<ReplayBundleVerifyResult>;

  putObject(bytes: Uint8Array, mediaType: string): Promise<ContentAddress>;
  getObject(address: ContentAddress): Promise<Uint8Array>;
  hasObject(address: ContentAddress): Promise<boolean>;
}

Storage rule:

Replay lookup key: request_hash + sequence_index
Object dedupe key: sha256(content bytes)

Pass criteria:

  • Current file-backed behavior remains unchanged.
  • Large observation bodies can later be moved to content-addressed objects without changing replay semantics.
  • Bundle verification can run through the store interface.
  • No hidden fallback to live upstream is introduced.

Server-Ready Architecture

Build the server path without forcing hosted adoption on day one. 

Local record:
agent -> local Datalox recording proxy -> real upstream MCP/API -> local bundle

Local replay:
agent -> local Datalox replay server -> verified local bundle

Fixture replay:
agent -> local Datalox replay server -> installed fixture pack bundle

Server registry:
local Datalox CLI/daemon -> Datalox server -> verified bundles, fixture packs, refs, metrics

The server should not be required for private internal API recording. It should become useful for team sharing, approvals, CI replay, fixture discovery, and cross-machine bundle pull.
