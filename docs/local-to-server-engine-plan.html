Local Engine To Server-Ready Plan

This plan defines what to do in datalox-api-gym after Option A: keep this repo as the local MCP VCR engine, make it plug-and-play, shape it for a future server registry, and consume curated world packs from the separate fixture repository.

Status: implementation and cleanup plan. Updated: 2026-05-20. Canonical boundary: project-definition.md, agentic-rl-layer-map.md, replay-bundle-schema.md, runtime-adapter-roadmap.html.

The product sentence should stay stable: Datalox turns real agent tool interactions into verified API worlds and grounded trajectory/eval exports.

This repo

datalox-api-gym is the engine.

It owns proxy record mode, replay mode, bundle packing, bundle verification, local storage, SDK entrypoints, CLI UX, server-shaped interfaces, and compact dataset exports derived from replay evidence.

API world repo

datalox-api-gym-worlds is the curated fixture library.

It already contains versioned public world packs for GitHub workflows, Slack support threads, search/RAG corpora, and Stripe-style SaaS API edge cases.

Future server

The hosted service is a verified replay and fixture registry first.

It stores, verifies, indexes, approves, shares, and serves bundles, world packs, world sets, and export manifests. It does not need to be the first recorder.

Decision

Do not turn this repo into a large fixture/data repository. This repo should remain the installable engine that can consume local private bundles, checked-in reference bundles, and later remote world packs from a registry.

The correct split is: 

datalox-api-gym  = runtime, CLI, SDK, store, verifier, MCP VCR, export adapters
datalox-api-gym-worlds = versioned public replay-backed world packs
Datalox server         = registry, auth, approvals, object storage, sharing, metrics

The engine must not depend on the API world repo or hosted server at runtime. The fixture repo and server should depend on the engine contract.

Updated framing for a real Agentic RL team: Datalox is not "a schema product." It is the tool-I/O evidence layer that gives rollout, eval, and data teams a pinned tool world plus grounded exports. The schema is necessary because it makes that workflow machine-checkable, but the wedge is reproducible tool behavior and exportable training views.

Product Boundary

What Datalox should do

  • Record real MCP/tool requests and observations.
  • Snapshot agent-visible MCP tools/list catalogs.
  • Pack verified replay_bundle.v1 artifacts.
  • Serve replay from bundle with upstream completely off.
  • Return deterministic replay misses for unseen requests.
  • Expose local CLI and SDK surfaces.
  • Expose server-shaped storage interfaces and APIs.
  • Measure replay hit rate, miss rate, tool rounds, latency, and cost metadata.
  • Derive task, trajectory, transition, reward, completion-frame, and split exports from replay evidence.

What Datalox should not do here

  • Do not build coding sandboxes or Docker environment rebuilders.
  • Do not build stateful behavioral simulators for unseen actions.
  • Do not write reward functions or judge agents.
  • Do not lead with schema as the main user-facing wedge.
  • Do not put a large curated fixture library in this repo.
  • Do not reintroduce the legacy wiki/note/promotion loop.

The precise wording is important: Datalox can serve recorded observations as record-based mocks. It does not construct open-ended behavioral mocks that respond faithfully to arbitrary unseen actions.

Real Agentic RL Data Jobs

A real Agentic RL team usually does not want "a bundle" as the final object. They want tasks, trajectories, step-level data, rewards, eval splits, or completion rows that can feed SFT, preference learning, reward modeling, regression tests, or online rollouts. The bundle is still useful, but as proof and replay source.

The ownership rule is:

Replay bundle = source evidence and replay proof
API world = pinned replay environment for an agent run
Trajectory = episode view over one task/run
Transition = step-level state/action/observation/reward view
Completion frame = exact model-visible prompt/response view
Reward record = versioned scoring source and reward vector
Split record = train/dev/test/replay partitioning

These objects may reference the same underlying tool content, but they must not all own it. Raw tool observations belong to replay evidence. Dataset views should reference that evidence and store only the extra semantics they own.

Object Owns Must reference Primary consumer
replay_bundle.v1 Exact tool I/O, MCP catalogs, checksums, redaction/export status. Source repo/session metadata. Replay, audit, fixture-world runtime.
API world Pinned activation of one or more verified replay bundles. Bundle ids and fixture metadata. Agent eval harnesses and CI.
Trajectory Task episode ordering, status, final answer, final reward. Task id, step ids, replay bundle id. Reward models, preference data, trajectory-level evals.
Transition One decision point: state ref, action, observation ref, step reward. Trajectory id and replay evidence refs. Credit assignment, process supervision, step evals.
Completion frame Exact model-visible context and response after context management. Transition id and raw observation refs. SFT and train/inference consistency checks.
Reward record Reward vector, scalar fold, source, version, normalization, clipping. Task, trajectory, or transition id. Reward models, RL, diagnostics.

Practical rule: replay bundle stores proof; all training datasets are views. If a view cannot point back to replay evidence or a declared reward source, it is not export-quality data.

Whole Environment Definition

A whole Agent RL environment is a task world with a stable reset/step protocol. It is larger than Datalox's current replay-backed API world because it must respond to new valid actions, not only replay observed tool calls. 

reset(task_id, seed) -> initial_observation
step(action) -> observation, reward, done, info

The minimum contract is:

Part Meaning Why it matters
Hidden state Ground-truth task state, runtime state, database state, browser state, or simulator state. The agent should not directly see answers or privileged fields.
Observation renderer Turns hidden state and tool results into the model-visible view. Train/inference consistency depends on what the policy actually saw.
Action schema Defines allowed tool calls, final-answer actions, ask-user actions, and forbidden operations. Invalid actions must produce learnable consequences instead of silent correction.
Transition engine Applies valid actions to the world and returns the next observation. This is what replay-backed fixtures lack for unseen paths.
Terminal logic Decides when the task is done and records the terminal reason. The model saying "done" is an action, not the environment's success judgment.
Reward/verifier hooks Compute outcome, process, constraint, cost, and anti-hacking signals from evidence. Reward must be traceable to environment evidence, not final prose alone.

Roadmap boundary:

Today:
  replay-backed API world
  = verified recorded tool/API observations
  = deterministic eval and debugging over observed scenarios

Later:
  runtime-backed whole environment
  = reset/step world backed by Docker, WebHarbor, CyberGym, E2B, Modal, OpenClaw, or a custom simulator
  = can handle new valid actions and still emit replay evidence

North Star Metric

The immediate product benchmark is Time To First Verified Replay. 

existing MCP server
  -> datalox proxy record
  -> replay bundle
  -> upstream off
  -> deterministic replay
  -> tamper check fails

The local demo target is under 2 minutes with the included fixture. The bring-your-own MCP target is under 5 minutes for a developer who already has an MCP server command. For Agentic RL data users, the follow-up benchmark is time from verified replay to grounded trajectory/transition export.

Ownership and anti-overlap rules:

  • tasks.jsonl owns task goal, initial input, allowed tools, gold constraints, and verifiability level.
  • trajectories.jsonl owns episode status, ordered step ids, final answer, and final reward refs.
  • steps.jsonl owns transition-level action, observation refs, and step reward refs.
  • completion_frames.jsonl owns model-visible prompt/context and response only.
  • rewards.jsonl owns reward source, version, reward vector, scalar fold, normalization, and clipping.
  • splits.jsonl owns train/dev/test/replay split assignment and leakage group ids.
  • Raw tool observations remain in replay bundles; export rows reference them by id/path/hash.

Keep heavy sandbox integration out of the core path until fixture replay and fixture sets are solid. The later runtime adapter work is specified separately in runtime-adapter-roadmap.html.

Boundary:

external runtime owns containers, services, filesystems, browsers, and verifiers
Datalox owns tool-I/O evidence, replay bundles, fixture activation, and exports

Pass criteria:

  • Runtime adapters are optional and do not make simple MCP/API fixture replay heavier.
  • Adapters capture tool_io_record.v1 evidence instead of replacing replay primitives.
  • Replay mode cannot silently start live upstream services.

Phase 8.6: Whole Environment Contract

Define the shape before implementing a full environment runtime. This prevents Datalox from accidentally mixing pure API Gym worlds with live simulator behavior.

Proposed schema:

environment_world.v1
  id
  version
  kind: replay_backed | runtime_backed | hybrid
  reset: task_id + seed -> observation
  action_schema_refs
  observation_schema_refs
  terminal_rules_ref
  reward_or_verifier_refs
  runtime_adapter_ref?
  replay_fixture_refs?
  export_policy

Pass criteria:

  • Replay-backed worlds cannot claim to handle unseen actions.
  • Runtime-backed worlds declare the external adapter that owns state transitions.
  • Every rollout emits tool-I/O evidence and can be packed into a replay bundle.
  • Terminal reasons and reward breakdowns are stored as versioned evidence, not hidden judge state.

Phase 10: Parallel Tool Interaction Support

Agentic RL papers increasingly optimize tool-use efficiency. Datalox should preserve enough structure to replay and measure parallel or batched tool rounds.

Add fields when schema migration is planned:

Pass criteria:

  • Sequential replay behavior remains unchanged.
  • Identical concurrent calls cannot be replayed by unstable arrival order.
  • If stable parallel slot metadata is missing, strict mode fails with an ambiguity error.

Server-Ready Architecture

Build the server path without forcing hosted adoption on day one. 

Local record:
agent -> local Datalox recording proxy -> real upstream MCP/API -> local bundle

Private API world:
local bundle -> private world pack -> datalox replay --fixture private-pack@version

World set replay:
agent -> local Datalox replay runtime -> pinned world set -> deterministic tool observations

Server registry:
local Datalox CLI/daemon -> Datalox server -> verified bundles, world packs, world sets, refs, metrics

The server should not be required for private internal API recording. It should become useful for team sharing, approvals, CI replay, fixture discovery, and cross-machine bundle pull.
