Datalox Briefing

Datalox should turn high-value traditional workflows into agent-native environments, then use replay evidence to make those environments auditable, reusable, and useful for training/eval teams. Recording is how private snapshots are authored, not the thing training teams consume. SFT and preference training are the practical first consumers; online RL is later, when reset/step environments and rollout throughput are mature.

From Yifan - Updated May 29, 2026 - Local draft, not yet public-facing copy

One-Sentence Positioning

Datalox reconstructs traditional human-facing workflows as agent-native domain environments, then packages replayable evidence and exports training/eval data for SFT, preference training, eval, and eventually RL.

This is intentionally not "Agentic RL infra first." RL is valuable, but it is expensive: rollouts are slow, verifiers are bottlenecks, and reward signals are often sparse. The near-term product value is agent-native workflow environments plus installable snapshot environments that produce clean trajectories for export.

Wedge Clarification

The business wedge should not be "replay company" in isolation. Replay is the evidence layer. The stronger wedge is making workflows that were built for humans usable by agents: structured tools, explicit state, deterministic validators, constrained app surfaces, and batchable task worlds. 

traditional app or workflow
  -> agent-native tools and state
  -> replayable tool I/O evidence
  -> deterministic verifier
  -> train/eval exports
LayerWhat It ProvesWhat Not To Claim
Agent-native domain environment A high-value workflow can be operated through typed tools, structured state, and deterministic validators instead of brittle GUI-only interaction. Do not claim every traditional app needs to be rewritten.
Replay evidence layer Training/eval teams can inspect exactly what the agent saw and replay recorded observations with no live fallback. Do not claim the full model trajectory is guaranteed to regenerate identically.
Export layer Verifier-passing trajectories can become SFT/eval/preference rows without trusting a one-off demo transcript. Do not lead with Datalox as a post-training platform.

Market Implication

Treat recent "RL environment company" and "RLaaS" commentary as a useful market signal, not as proof that Datalox should become an RLaaS company. The signal is that labs and later enterprises may pay recurring spend for high-quality agent experience capacity: real tasks, verified outcomes, reproducible traces, and data that training teams can trust. The Felicis RL infrastructure overview is a useful public framing for this split between environments, RLaaS, traces, evaluation, and data capacity.

Hugging Face-style openness changes the threat model. The risk is not "open source kills Datalox." Open artifacts can increase adoption and create the distribution layer Datalox should publish into. The risk is narrower: if an open platform standardizes replay evidence, verifier metadata, export gates, and private snapshot authoring, then Datalox should not fight that standard. Datalox should implement it better and compete on engineering execution: sandbox/runtime composition, capture reliability, deterministic replay guarantees, verifier authoring, private workflow packaging, data QA, and domain-specific fixture worlds. See the Hugging Face Hub docs and Spaces docs for the open distribution pattern.

Exclusive lab relationships can look like services revenue instead of scalable infrastructure. Keep fixture worlds portable: local replay, Hugging Face-compatible exports, and training-stack-agnostic rows. Do not design the system around one lab, one training stack, or one private stack.
RLaaS may become the enterprise training layer as more companies post-train agents. Integrate underneath RLaaS and training teams as the verified environment/data supplier. Do not compete with full post-training orchestration before the replay/data layer is proven.
Open platforms can standardize artifact formats while still leaving room for strong engineering products. Adopt open schemas when they win, publish Datalox artifacts to open hubs, and compete on higher-quality capture, replay, verification, sandbox composition, and private/domain workflow production. Do not depend on a proprietary generic environment format as the moat.
Positioning guardrail: Datalox should sound like "agent-native workflow environments with verified replay evidence," not "another RL environment platform" and not "managed RL training." If a prospect already has a sandbox, training stack, and reward code, Datalox should still be useful because it provides the domain workflow surface and versioned evidence layer those systems can consume. If Hugging Face or another open platform owns the distribution standard, Datalox should be the best system for producing, verifying, and operating compatible private/domain fixture worlds.

Current Implemented Surface

The repo now has a concrete first path for model-server users instead of only fixture metadata: 

OpenAI-compatible model endpoint, including vLLM
  -> datalox run
  -> replay fixture world
  -> datalox_run.v1
  -> datalox export sft
  -> sft_frame.v1 JSONL

This is the practical bridge to SFT-first training teams. Datalox does not replace vLLM, SGLang, LiteLLM, verl, or a lab's internal training stack. It provides the declared replay world, grounded transcript, and export row. 

datalox run \
  --fixture-set support-triage-basic@2026-05.0 \
  --base-url http://127.0.0.1:8000/v1 \
  --model Qwen/Qwen3-1.7B \
  --api-key token \
  --prompt "Resolve this support triage task." \
  --out runs/support-triage-qwen \
  --json

datalox export sft \
  --run runs/support-triage-qwen \
  --out exports/support-triage-qwen.sft.jsonl \
  --json

Implemented now: OpenAI-compatible model adapter, replay tool runtime, datalox_run.v1, datalox run, sft_frame.v1, and datalox export sft. Still next: published serious fixture sets, completion frame export, preference pair export, direct verl adapter, and live domain environment rollout proofs.

Who Uses What

Rollout / Environment Owner

Runs agents against real tools, domain MCPs, APIs, or app worlds. Uses Datalox to author private snapshots, package replay bundles, and publish versioned fixture worlds.

Training / Eval Consumer

Wants installable fixture sets and clean versioned exports, not capture infrastructure. Starts with SFT and preference data; may later use reward/eval rows and online RL.

Fixture / Data Provider

Curates approved snapshot fixture packs and fixture sets so downstream teams can train, evaluate, and compare agents without live tool credentials.

Authoring Path vs Consumption Path

Authoring path  (rollout owner - needs live tools)
live domain MCP / API / app world
  -> Datalox snapshot authoring path
  -> tool_io_record.v1
  -> replay_bundle.v1
  -> fixture pack
  -> fixture set

Consumption path  (training/eval consumer - no live tools)
versioned API/MCP snapshot fixture set install
  -> replay runtime
  -> datalox_run.v1
  -> sft_frame.v1 now
  -> completion_frame.v1 + preference_pair.v1 next
  -> SFT / preference training / reward modeling / eval / regression
  -> online RL later, only with runtime-backed reset/step

Important boundary: replay mode never invents observations and never falls back to live tools. Unseen calls return structured replay misses.

Training Consumer Order

OrderConsumerDatalox OutputBoundary
1 SFT-first teams sft_frame.v1 now, completion_frame.v1 next Use successful runs first. Recovery trajectories can be added when labeled. Do not blindly clone failures.
2 Preference training preference_pair.v1 next Success > recovery > failure under the same task, all backed by replay evidence.
3 Reward modeling / eval reward_record.v1, verifier provenance, reference scheme Datalox records evidence and reference schemes. Training teams own reward design.
4 Online RL transition_frame.v1, runtime-backed rollout evidence Requires reset/step worlds, rollout throughput, and verifier capacity.

Reward Boundary

Datalox should not own canonical reward design. Reward is task-specific and belongs to the team doing training. Datalox should preserve enough evidence so the reward can be recomputed, audited, or replaced. 

Datalox owns:
  reward source
  reward version
  verifier command/result
  evidence refs
  optional reference reward scheme

Training team owns:
  final reward function
  weights
  normalization
  clipping
  training objective

Why SFT First

Zheng's feedback changes the priority order but not the product boundary: online Agent RL is expensive, slow to roll out, and often suffers from sparse reward. Many teams will get value sooner from environments that generate many clean trajectories for SFT and preference training.

Datalox should therefore optimize the first mile: 

fixture/world breadth
  -> stable tool observations
  -> model-visible transcript
  -> exportable SFT/completion/preference rows
  -> model-team-owned optimization

This is not a retreat from RL. It is the path that produces useful data before a team can afford large on-policy rollout loops.

Environment Strategy

A single deep environment proves seriousness, but breadth is likely better for improving general agent capability. The recommended strategy is: 

one credible deep-enough proof
  -> many shallow/medium verified sub-environments
  -> SFT/preference exports at scale
  -> runtime-backed online RL later

For bio, this means an environment family rather than one perfect simulator.

Runtime Ownership Split

The right split is not "adapter only" and not "Datalox becomes a generic sandbox company." Datalox should own sandbox-backed environments when the state substrate is part of the product, especially constrained scientific workspaces. It should integrate with external whole-environment runtimes when those systems already own a mature sandbox, app state, verifier, and task-generation stack. 

Datalox Agent Replay core
  = replay bundles, fixture sets, deterministic replay, grounded exports

Datalox-owned domain/runtime worlds
  = constrained file-backed task environments with tools, revisions,
    validators, reset metadata, and replay evidence

External whole-environment runtimes
  = OpenComputer, AppWorld, CyberGym, WebHarbor, E2B, Docker, or lab runtimes
  = integrate first through explicit runtime-backed record/rollout paths

Product rule: build Datalox-owned environments where Datalox owns the domain state and validator contract. Use adapters where another system already owns the sandbox/runtime semantics. In both cases, every useful rollout should emit replay evidence before export.

AppWorld Lesson

AppWorld is a strong reference for what a whole environment looks like: simulated users, app databases, APIs, task generation, state-based evaluation, checkpointing, logs, and an MCP/API surface.

The lesson is not "clone AppWorld." The practical Datalox proof is: 

AppWorld MCP/API server
  -> Datalox authors a versioned snapshot
  -> replay_bundle.v1
  -> fixture set
  -> SFT/preference/eval exports

OpenComputer Lesson

OpenComputer and the OpenComputer repo are a strong reference for verifiable computer-use worlds: desktop app sandboxes, generated tasks, app-specific state verifiers, trajectory logs, screenshots, and partial-credit rewards.

The lesson is not "clone OpenComputer's full desktop stack." The practical Datalox proof is: 

OpenComputer task run
  -> capture screenshots, actions, verifier outputs, and reward evidence
  -> replay_bundle.v1
  -> fixture set
  -> completion_frame.v1 / preference_pair.v1 / reward_record.v1 exports

OpenComputer is the external-runtime side of the split. FlowCyto and the bio environment family are the Datalox-owned domain-environment side.

Reference Papers And Systems

These references are not competitors to copy one-for-one. They define the environment design space Datalox should sit under: state substrates, action APIs, observation renderers, verifiers, rollout logs, and exportable training/eval evidence.

ReferenceWhat It ShowsDatalox Implication
Agent-World Environment scaling through stateful databases, executable tool ecosystems, task synthesis, verifier scripts, and multi-environment training. A useful world is not one tool. A fixture set should represent a tool ecosystem plus state, task specs, verifier refs, and rollout evidence.
EvoEnv Environment-as-code: reusable executable artifacts that generate instances, compute references, and score responses after staged validation. Later domain environments should expose deterministic generation and verification hooks. For now, Datalox should snapshot and export verified rollouts.
HyperEyes Search-agent training where action-space design and efficiency-aware rewards matter as much as runtime infrastructure. Datalox should preserve tool-call cost, turn count, replay misses, and verifier evidence so training teams can build their own efficiency signals.
AppWorld paper
AppWorld repo 		Whole app-state world: apps, APIs, users, databases, task setup, execution logs, state-based evaluation, and MCP support. Near-term proof should be an adapter: AppWorld rollout -> Datalox replay bundle -> fixture set -> SFT/eval export, not a clone of AppWorld.
OpenComputer paper
OpenComputer repo 		Verifier-grounded computer-use worlds: desktop app sandboxes, generated tasks, state inspection endpoints, trajectory logs, screenshots, and auditable partial-credit rewards. Treat as a mature external whole-environment runtime to integrate with first. Datalox should capture and package its run evidence, not clone the generic desktop sandbox stack.
CyberGym paper
CyberGym repo 		Heavy executable benchmark environment for real vulnerability tasks, including large benchmark data, Docker/server execution, and PoC evaluation. Datalox should not own CyberGym-style runtime distribution. It can record and replay agent-visible interactions around such runtimes.
WebHarbor repo Browser-world direction: local website mirrors, resettable state, UI/task review, and browser-agent execution surfaces. Future runtime adapters may launch WebHarbor-style worlds, while snapshot replay remains finite and replay-miss based.

Common pattern across the references: state substrate + action API + observation renderer + verifier + rollout evidence. Datalox's first business wedge should be selected agent-native domain environments where we own the state/action/verifier substrate; the evidence/replay/export layer is what makes those environments useful to training and eval teams.
