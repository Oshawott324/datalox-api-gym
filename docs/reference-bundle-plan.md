# Reference Replay Bundle Plan

Status: active reference set.
Updated: 2026-05-20.

This plan defines the first proof artifact for Datalox API Gym: a small
set of public, verified replay bundles that show the tool-I/O VCR loop working
end-to-end.

Canonical boundary:

- [agentic-rl-layer-map.md](./agentic-rl-layer-map.md)
- [replay-bundle-schema.md](./replay-bundle-schema.md)
- [tool-io-store-schema.md](./tool-io-store-schema.md)

## Goal

Maintain 3 excellent reference replay bundles before trying to publish a larger
collection.

The point is not volume. The point is proof:

```text
agent tool call -> tool_io_record.v1 -> replay_bundle.v1 -> deterministic replay -> optional derivatives
```

A good reference bundle lets another engineer inspect exactly what happened,
verify checksums locally, and replay recorded tool observations with the live
upstream server turned off.

## Product Boundary

Datalox owns the tool-I/O record/replay layer.

It records real agent-visible tool observations once, packages them into
verifiable replay bundles, and replays those observations deterministically
later.

Datalox does not own:

- sandbox runtimes
- environment construction
- behavioral mock construction
- reward functions
- judge agents
- RL training algorithms
- hidden chain-of-thought capture

Reference bundles should demonstrate this boundary. They can include
provenance for runtimes, verifiers, or environments when strict schemas accept
it, but they must not imply that Datalox runs the sandbox or computes reward.

## First Milestone

The first 3 bundles are:

1. `ref-mcp-success`
   - one MCP tool call
   - one successful structured observation
   - bundled `mcp_tool_catalog.v1`
   - replay succeeds with upstream disabled

2. `ref-mcp-repeated-call`
   - same tool name and arguments called at least twice
   - records use the same `request_hash`
   - records use increasing `sequence_index`
   - replay returns observations in the original order

3. `ref-mcp-error-observation`
   - one upstream tool error or `isError` style observation
   - error is represented structurally
   - replay returns the recorded error observation without calling upstream

These should use safe, synthetic, publishable tool calls. Do not use private
customer traces for the first public references.

## Bundle Requirements

Each reference bundle must contain:

- `manifest.json`
- `checksums.json`
- at least one `tool_io_record.v1`
- one `mcp_tool_catalog.v1` when captured through the MCP proxy
- optional `agent_turn.v1` context when useful
- explicit export/redaction status

Each reference bundle must pass:

```bash
npm run build
node dist/src/cli/main.js bundle verify --repo . --bundle .datalox/replay-bundles/<bundle-id> --json
```

Do not put ad hoc README files inside the sealed bundle unless the strict bundle
schema accepts them. Public explanation belongs beside the bundle, for example:

```text
examples/reference-bundles/<bundle-id>/README.md
.datalox/replay-bundles/<bundle-id>/
```

## Technical Workflow

The checked-in reference set is generated by:

```bash
node examples/reference-bundles/generate-reference-bundles.mjs --build
```

The lower-level workflow is:

### 1. Prepare a safe upstream MCP fixture

The fixture should expose a small deterministic tool catalog:

```text
tools/list -> one or two simple tools with inputSchema and outputSchema
tools/call -> structured success or structured error observations
```

The fixture must not depend on secrets, live APIs, private files, or unstable
network state.

### 2. Record through the MCP VCR proxy

Start the proxy in record mode:

```bash
node dist/src/cli/main.js proxy \
  --mode record \
  --repo . \
  --config examples/mcp-proxy/<fixture>.record.json \
  --json
```

Then point an MCP client at the proxy and call the fixture tools.

Expected recorded artifacts:

```text
.datalox/tool-io/records/*.json
.datalox/mcp-tool-catalogs/*.json
```

### 3. Pack the replay bundle

```bash
node dist/src/cli/main.js bundle pack \
  --repo . \
  --bundle-id <bundle-id> \
  --json
```

Expected bundled artifacts:

```text
.datalox/replay-bundles/<bundle-id>/manifest.json
.datalox/replay-bundles/<bundle-id>/tool-io/*.json
.datalox/replay-bundles/<bundle-id>/mcp-tool-catalogs/*.json
.datalox/replay-bundles/<bundle-id>/checksums.json
```

### 4. Verify the bundle

```bash
node dist/src/cli/main.js bundle verify \
  --repo . \
  --bundle .datalox/replay-bundles/<bundle-id> \
  --json
```

Verification must fail if a checksum changes, a listed file is missing, an
unlisted file appears, or a bundled MCP tool catalog is invalid.

### 5. Replay with upstream disabled

Start replay mode against the verified bundle:

```bash
node dist/src/cli/main.js proxy \
  --mode replay \
  --repo . \
  --bundle .datalox/replay-bundles/<bundle-id> \
  --json
```

Then call the same tool requests through the proxy.

Pass criteria:

- replay starts without the upstream MCP fixture running
- `tools/list` is served from the bundled `mcp_tool_catalog.v1`
- each `tools/call` resolves by `request_hash + sequence_index`
- returned observations match the recorded observations
- missing calls return a deterministic replay miss
- no live upstream fallback occurs

## Hard Gates

A reference bundle does not ship unless:

- `npm test` passes
- `npm run check` passes
- `git diff --check` passes
- `bundle verify` passes for the bundle
- replay works with upstream disabled
- all records have explicit export/redaction status
- no secrets, credentials, PII, or private customer data appear
- the public README explains the task, tool calls, expected replay behavior,
  verification command, and schema references

## Publication Shape

For the first 3 bundles, prefer a GitHub-native layout:

```text
examples/reference-bundles/
  ref-mcp-success/
    README.md
  ref-mcp-repeated-call/
    README.md
  ref-mcp-error-observation/
    README.md

.datalox/replay-bundles/
  ref-mcp-success/
  ref-mcp-repeated-call/
  ref-mcp-error-observation/
```

After the first 3 bundles are stable, mirror them as GitHub release artifacts.
Only consider Hugging Face Datasets after there are enough bundles to browse as
a collection. The first milestone is a proof artifact, not a dataset launch.

## Future Schema Extensions

These are useful, but they should remain future strict-schema work until the
code accepts them:

- model/runtime version
- sampling configuration
- system prompt hash
- tool version and corpus snapshot
- sandbox image digest
- verifier command and version
- reward source and reward version

Do not add loose `runtime`, `env`, `reward`, or `metadata` bags to
`replay_bundle.v1`. If the information matters before schema support exists,
store it in a validated artifact or in derivative provenance that clearly
references the source replay bundle.

Derivative labels such as `trajectory_type`, `first_wrong_step`, and
`replay_bundle_ref` belong to derivative trajectory rows, not the replay bundle
itself.

## Two-Week Plan

Current status: the first bundle set exists and is regression-tested. The next
two-week plan is for hardening and publication polish.

Week 1:

1. Review the generated bundle READMEs.
2. Run the generator on a clean checkout.
3. Confirm the bundle checksums are stable enough for release artifacts.
4. Confirm no private paths, secrets, PII, or customer data appear.
5. Use the bundles in at least 3 design-partner conversations.

Week 2:

1. Verify replay with upstream disabled on another machine or fresh clone.
2. Run full repo checks.
3. Review redaction/export state.
4. Publish the bundles in the repo.
5. Use the bundles in design-partner conversations.

## Discovery Use

Use the bundles to ask sharper buyer questions:

```text
For the scenarios you already observed, would this kind of verified replay
bundle remove repeated rollout or mock-rebuild work?
```

If the buyer needs unseen stateful behavior, Datalox is complementary but not
the full solution. If the buyer needs scenario reproduction, eval replay, or
tool/API record-based mocking, Datalox is directly relevant.
